From server/routes/me.ts (PATCH-shaped error handling for unique constraint violations):
```typescript
try {
  const [updated] = await db.update(users)...
} catch (err) {
  const code = (err as { code?: string }).code;
  if (code === '23505') return c.json({ error: 'taken' }, 409);
  throw err;
}
```
PATCH /api/cities/:id won't trigger 23505 in normal usage (we don't expose order_index editing here — that's 05-03's reorder). But the pattern for catching pg error codes is the canonical reference.

Drizzle transaction shape (used by POST and 05-03's reorder):
```typescript
const inserted = await db.transaction(async (tx) => {
  const [maxRow] = await tx.select({ max: sql<number>`COALESCE(MAX(${cities.orderIndex}), -1)` })
    .from(cities)
    .where(eq(cities.userId, me.id));
  const nextIdx = (maxRow?.max ?? -1) + 1;
  const [row] = await tx.insert(cities).values({
    userId: me.id,
    orderIndex: nextIdx,
    ...validated, // from Zod parse
  }).returning();
  return row;
});
```
Critical: BOTH operations use `tx`, not `db`. Drizzle's transaction binds them to the same connection. The deferrable unique constraint (DATA-02) checks at COMMIT, so even if two concurrent transactions race, the second to COMMIT will fail cleanly with 23505 — the first wins. Catch 23505 and either retry once OR return 409 with copy "Try again — please retry your save."

    ```typescript
    // contract. Do NOT change to expect(both).toBe(201).
    ```

    ```typescript
    citiesRouter.patch('/:id', async (c) => {
      const me = c.var.user;
      const id = c.req.param('id');
      const raw = await c.req.json().catch(() => null);
      const parsed = updateCitySchema.safeParse(raw);
      if (!parsed.success) {
        return c.json({ error: 'invalid_input', issues: parsed.error.issues }, 422);
      }

      try {
        // Build the update object from validated keys only — never spread raw.
        const patch: Record<string, unknown> = { updatedAt: new Date() };
        for (const [k, v] of Object.entries(parsed.data)) {
          if (v !== undefined) patch[k] = v;
        }

        const [updated] = await db.update(cities)
          .set(patch)
          .where(and(eq(cities.id, id), eq(cities.userId, me.id)))
          .returning();
        if (!updated) return c.json({ error: 'not_found' }, 404);
        return c.json(updated);
      } catch {
        // Malformed UUID etc — collapse to 404 to avoid leaking DB errors.
        return c.json({ error: 'not_found' }, 404);
      }
    });
    ```

    ```typescript
    citiesRouter.delete('/:id', async (c) => {
      const me = c.var.user;
      const id = c.req.param('id');
      try {
        const result = await db.delete(cities)
          .where(and(eq(cities.id, id), eq(cities.userId, me.id)))
          .returning({ id: cities.id });
        if (result.length === 0) return c.json({ error: 'not_found' }, 404);
        c.status(204);
        return c.body(null);
      } catch {
        return c.json({ error: 'not_found' }, 404);
    ```

    **A note on `order_index` and DELETE:** when a row is deleted, the remaining rows have a hole in their order_index sequence (e.g., 0, 1, 3 instead of 0, 1, 2). This is FINE. Don't try to compact the sequence on delete; you'd just create a new race window. Rely on the reorder endpoint (05-03) for any explicit ordering changes.
