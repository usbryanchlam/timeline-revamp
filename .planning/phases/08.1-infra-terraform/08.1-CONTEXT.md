## Phase Boundary

Phase 08.1 replaces the manual, click-trail OCI provisioning documented in Phase 8 with Terraform-managed Infrastructure-as-Code. **Inserted between Phase 8 and Phase 9** (decimal phase, INSERTED marker). Phase 8 itself is currently paused at Wave 3 (DNS cutover + iPhone UAT); 8.1 lands fully before that resumes, and the live DNS flip then targets the TF-provisioned VM rather than a hand-built one.

**What 8.1 delivers:**

1. **Terraform stack** at `infra/terraform/` covering the full OCI footprint: VCN + subnet + internet gateway + route table + security list (80/443/22), Ampere A1 compute (`VM.Standard.A1.Flex`, 4 OCPU / 24 GB free-tier max), reserved public IP, cloud-init `user_data` that replicates `infra/setup.sh` exactly, Object Storage photos bucket + S3-compat CORS rules, IAM dynamic group + policy for Instance Principal access to the photos bucket from the VM.
2. **Remote state backend** in an OCI Object Storage bucket (S3-compat endpoint) with Terraform 1.10+ native conditional-write locking (`use_lockfile = true`) — no DynamoDB analog needed.
3. **GitHub Actions CI/CD** with OIDC trust between GitHub and OCI (no long-lived API keys in GHA secrets), `plan` on PR with output comment, `apply` on merge to `main` gated by a `production` GitHub Environment requiring manual approval, plus a weekly drift-detection cron that opens a GitHub Issue if `terraform plan -detailed-exitcode` shows drift.
4. **Deletion of `infra/setup.sh`** — cloud-init YAML absorbs every line. `infra/DEPLOY.md` updated accordingly (TF provisioning prepended; manual sections that 8.1 replaces are removed; Nginx/Let's Encrypt/DNS Cutover/Smoke Test sections from 08-01/02/03 stay verbatim — those are operator steps that run AFTER `terraform apply`).

## Implementation Decisions

### Already-locked decisions (entered the workflow as priors)

- **D-01:** **Provider:** Official `oracle/oci` Terraform provider. NOT OpenTofu, NOT third-party. Pin `~> 6.0` or current latest stable as of 8.1 plan time (planner verifies at plan-phase).
- **D-02:** **State backend:** Remote in OCI Object Storage via the S3-compat endpoint. Backend `s3` with `endpoints.s3 = "https://<namespace>.compat.objectstorage.<region>.oraclecloud.com"`. Authentication via OCI Customer Secret Key (one-time, stored in GHA secrets) for the state backend itself; OIDC for everything else.
- **D-03:** **State locking:** Terraform 1.10+ native conditional-write locking (`use_lockfile = true` on the `s3` backend). Requires `terraform >= 1.10.0`. OCI S3-compat API supports `If-None-Match`; no DynamoDB analog needed. Verified by Terraform 1.10 release notes (Nov 2024).
- **D-04:** **GHA → OCI auth:** OIDC trust. OCI IAM Identity Provider for GitHub Actions; trust policy bound to repo + branch (see D-15). No long-lived OCI API keys in GitHub Secrets except the one-time state-backend Customer Secret Key.
- **D-05:** **Photos identity:** Instance Principal on the Ampere A1 VM. OCI Dynamic Group matches the instance's OCID; IAM policy grants `manage objects in compartment <id> where target.bucket.name = 'timeline-photos'`. SDK auto-detects on the VM via instance metadata endpoint; no API keys in `.env`.

### TF code layout + cloud-init (Area 1)

- **D-06:** **Flat single-stack layout** at `infra/terraform/`, one file per concern: `main.tf`, `providers.tf`, `variables.tf`, `outputs.tf`, `network.tf`, `compute.tf`, `storage.tf`, `iam.tf`, `cloud-init.tf`. No `modules/` subdirectory. Rationale: single-VM single-bucket single-environment project; modules add abstraction overhead with no reuse payoff.
- **D-07:** **Delete `infra/setup.sh`** — cloud-init YAML (`infra/cloud-init.yaml`, rendered as `templatefile()` in `cloud-init.tf`) absorbs every line. Mapping:
  - `apt-get update && upgrade` → cloud-init `package_update: true` + `package_upgrade: true`
  - apt install list → `packages:` array (docker.io, docker-compose-plugin, nginx, certbot, python3-certbot-nginx, iptables-persistent, git)
  - `usermod -aG docker ubuntu` → `groups:` and `users:` mapping
  - iptables ACCEPT-before-REJECT at position 6 → `runcmd:` block with the exact `iptables -I INPUT 6 ...` invocations + `netfilter-persistent save`
  - `timedatectl set-timezone UTC` → `timezone: UTC`
  - `mkdir /opt/timeline-revamp; chown ubuntu:ubuntu` → `runcmd:` (or `write_files:` placeholder + chown)
  - `git clone https://github.com/usbryanchlam/timeline-revamp.git /opt/timeline-revamp` → `runcmd:` `sudo -u ubuntu git clone ...`
  - `mkdir /opt/timeline-revamp/.oci; chmod 700` → `runcmd:`
- **D-08:** **`.env` delivery unchanged from Phase 8 D-13** — operator scp's `.env` after `terraform apply` succeeds. TF does not handle secrets. Cloud-init writes a small `/opt/timeline-revamp/.env.example.placeholder` reminder file pointing at `infra/DEPLOY.md` "Step 3: Provision secrets". Rationale: keeps secrets out of TF state, out of Object Storage, off the laptop-to-Vault path. Adds friction (`scp` after apply) but matches existing 08-CONTEXT D-13 and avoids expanding the secret surface area for v1.

### State bucket bootstrap + region/shape (Area 2)

- **D-09:** **State bucket bootstrap = one-time `oci os bucket create` CLI**, documented in `infra/DEPLOY.md` § "Bootstrap (one-time)". Steps:
  1. `oci os ns get` → capture `namespace`
  2. `oci os bucket create --name timeline-tfstate --compartment-id <compartment_ocid> --versioning Enabled`
  3. Set retention rule via OCI CLI (or via console): retain deleted objects 90 days
  4. Generate a Customer Secret Key for state-backend auth: `oci iam customer-secret-key create --user-id <user_ocid>` → store access-key + secret-key in GHA secrets `OCI_S3_ACCESS_KEY` / `OCI_S3_SECRET_KEY`
  5. Initialize: `cd infra/terraform && terraform init -backend-config="bucket=timeline-tfstate" -backend-config="region=<region>" -backend-config="endpoints={s3=\"https://<namespace>.compat.objectstorage.<region>.oraclecloud.com\"}" ...`
  No bootstrap TF module, no shell wrapper. ~5 documented commands beats either alternative for a one-and-done bring-up.
- **D-10:** **State bucket settings:** versioning **ON**, encryption (default), 90-day retention on deleted objects. Versioning recovers from accidental `terraform state rm` or corrupt apply. Retention covers the typical "oh-no" recovery window.
- **D-11:** **Region:** Use the user's existing free-tier home region (set as TF variable `region`, value supplied via `terraform.tfvars`, NOT committed). Planner adds a `variables.tf` definition with no default, forcing explicit declaration. Availability Domain: planner picks the AD that has free-tier A1 capacity at plan time (often AD-1; OCI capacity varies).
- **D-12:** **Compute shape:** `VM.Standard.A1.Flex`, **4 OCPU / 24 GB RAM** (free-tier maximum). Phase 8 D-08 specified ≥2/8 minimum; 8.1 goes to free-tier max for headroom (Postgres + future MP4 worker + image build CPU spikes). Boot volume: 100 GB (free-tier allows up to 200 GB; 100 covers Docker images + Postgres data + photo cache with margin). Image: latest Canonical Ubuntu 22.04 LTS aarch64 (looked up via `oci_core_images` data source filtered by `operating_system = "Canonical Ubuntu"` + `operating_system_version = "22.04"` + `shape = "VM.Standard.A1.Flex"`).

### Photos bucket disposition (Area 3)

- **D-13:** **Existing photos bucket from Phase 6 dev contains only throwaway test data** — safe to destroy before first `terraform apply`. `infra/DEPLOY.md` § "Bootstrap" includes a "delete pre-existing photos bucket (only if it has no production data)" step before bootstrap proceeds. Alternative: `terraform import oci_objectstorage_bucket.photos <ocid>` if the operator prefers idempotent adoption; planner's discretion at plan time.
- **D-14:** **Photos bucket name:** `timeline-photos` (no environment prefix). Single-environment project; OCI bucket names are namespace-scoped so no global-uniqueness collision risk. If a staging environment lands later, add `var.environment` prefix at that point.
- **D-15:** **CORS rules replicate Phase 6 exactly via `null_resource` + `aws s3api put-bucket-cors`.** The `oci_objectstorage_bucket` provider resource does NOT support a native `cors_rules` nested block (verified in RESEARCH Pattern 4 — supersedes the original D-15 wording). Configure via `null_resource.photos_cors` with a `local-exec` provisioner that shells out to `aws s3api put-bucket-cors --endpoint-url <s3-compat-url> --bucket timeline-photos --cors-configuration file://<rendered-json>`. Set `AWS_REQUEST_CHECKSUM_CALCULATION=when_required` in the local-exec env block per RESEARCH Pitfall 9. Phase 6's exact CORS values extracted at plan-phase time via `oci os bucket get-bucket-cors-policy` or `aws s3api get-bucket-cors`.
- **D-16:** **PARs (Pre-Authenticated Requests) NOT managed by TF.** PARs are runtime artifacts created on-demand by the API per upload. They have expirations and per-object scope; codifying them in TF would create thousands of stale resources. PROJECT.md locked PARs as the upload mechanism in Phase 6; that stands.

### GHA workflow + drift detection (Area 4)

- **D-17:** **Apply gate: GitHub Environments manual approval.** Workflow structure:
  - `pull_request` trigger → job `plan`: `terraform plan -no-color -out=tfplan` → posts plan output as PR comment via `actions/github-script` or `dflook/terraform-plan` action.
  - `push: branches: main` trigger → job `plan` (re-runs against main) → job `apply` with `environment: production` (requires `concurrency: terraform-apply` lock + required reviewer = user). The `production` Environment in GitHub repo settings has user listed as a required reviewer; the apply job pauses until approved in the Actions UI.
- **D-18:** **Drift detection: weekly cron** (`schedule: - cron: '0 12 * * 1'` Monday noon UTC) runs `terraform plan -detailed-exitcode`. Exit code 2 (changes detected) triggers an `actions/github-script` step that opens a GitHub Issue titled `[drift] Terraform plan detected unexpected changes` with the plan output in the body and labels `drift, infrastructure`. Free; no external service required. If an issue with that title is already open, append to it instead of creating duplicates (idempotent).
- **D-19:** **OIDC trust scoping:** Tight on `main` branch. OCI IAM Identity Provider trust policy:
  - **Main (apply) identity:** `subject = "repo:usbryanchlam/timeline-revamp:ref:refs/heads/main"`, OCI policy grants full `manage` on the timeline compartment (TF needs to create/destroy resources).
  - **PR (plan) identity:** Planner's discretion at plan-phase time. Two viable paths:
    - (a) Single tight identity → PR workflows skip `terraform plan` against real OCI; run `terraform fmt -check` + `terraform validate` only. PR reviewers don't see plan output until merge.
    - (b) Separate read-only identity scoped to `repo:usbryanchlam/timeline-revamp:pull_request` (or `repo:usbryanchlam/timeline-revamp:ref:refs/pull/*/merge`) with OCI policy `inspect all-resources in compartment` + `read object-family in compartment` (read-only). PR workflows run `terraform plan` against real tenant; reviewers see actual diff.
  - **Default to (b)** unless the planner finds OIDC subject syntax limitations that make it impractical — actual plan visibility on PRs is the more valuable behavior. The "tight on main" choice covers the write path; PR read access is additive defense-in-depth.

### Claude's Discretion

- **OCI compartment layout** — root compartment OK for v1; planner may propose a child `timeline-prod` compartment for future blast-radius isolation. Use a TF variable `compartment_ocid` either way.
- **Defined tags vs free-form tags** — defined-tag namespace `timeline-revamp` with keys `phase`, `managed-by`, `environment` if the planner finds value (e.g., for cost-tracking reports). Free-form tags acceptable for v1.
- **Provider version pin policy** — `~> 6.0` (allow patch updates) vs. exact pin `= 6.x.y`. Planner picks; Dependabot/Renovate setup is out of scope for 8.1 (Phase 9+ concern).
- **State backend Customer Secret Key rotation** — manual for v1. Document the procedure in `infra/DEPLOY.md` § "Operations". Automated rotation is out of scope.
- **PR-on-fork OIDC** — if any external contributor submits a PR from a fork, the OIDC trust won't fire (forks don't get the OIDC token). Plan workflows fall back to local-only validation. Document briefly; not a v1 concern (solo project).
- **`infra/DEPLOY.md` rewrite shape** — keep the existing `## Nginx + Let's Encrypt`, `## DNS Cutover`, `## Smoke Test` sections verbatim (those are still operator steps after `terraform apply`). Prepend a `## Terraform Provisioning` section. Remove the old `## OCI VM Setup` section (cloud-init replaces it). Planner finalizes section ordering.
