# Phase 08.1: infra-terraform - Research

**Researched:** 2026-05-15
**Domain:** Infrastructure-as-Code via Terraform on OCI (Ampere A1 + Object Storage + IAM) with GitHub Actions OIDC CI/CD
**Confidence:** MEDIUM-HIGH (TF/OCI provider syntax HIGH; S3-compat backend on OCI MEDIUM with known compat regression; GHA→OCI OIDC MEDIUM — relies on Identity Propagation Trust which requires an Identity Domain; CORS-via-TF LOW — provider does NOT expose `cors_rules`)

## Summary

Phase 08.1 introduces the project's first Terraform stack. The grain of the work is small (one VM, one bucket, one IAM dynamic group + policy, one OIDC trust, one GHA workflow), but four landmines need surfacing before the planner writes plans:

1. **`oci_objectstorage_bucket` has NO `cors_rules` argument.** D-15 in CONTEXT.md assumed it did. The official Terraform Registry argument reference lists `access_type`, `auto_tiering`, `bucket_scope`, `defined_tags`, `freeform_tags`, `kms_key_id`, `metadata`, `object_events_enabled`, `storage_tier`, `versioning`, and `retention_rules` — **no `cors_rules`**. The OCI Native API itself does not accept `corsRules` (the project's `feedback_oci_cors_via_s3.md` memory documents this). CORS must be applied out-of-band via the AWS-S3-compat endpoint. Two viable shapes: (a) `null_resource` + `local-exec` calling `aws s3api put-bucket-cors --endpoint-url <s3-compat>` after bucket creation, or (b) an operator runbook step that runs the same `aws s3api` command post-`terraform apply`. Both keep TF declarative for the bucket itself and pragmatic for the CORS overlay.

2. **Terraform 1.10.x is the safe pin; 1.11.2 is BROKEN with `use_lockfile` against OCI.** Issue `hashicorp/terraform#36742` documents that 1.11.2's S3 backend sends `x-amz-content-sha256` headers that OCI's S3-compat endpoint rejects with `InvalidArgument: x-amz-content-sha256 must be UNSIGNED-PAYLOAD or a valid sha256 value`. Pin to `>= 1.10.0, < 1.11.0` for now, or test 1.11.3+ against OCI before pinning. **There is also a NEW NATIVE `backend "oci"` shipping with Terraform 1.12+** (announced on the Oracle Cloud Infrastructure blog) that supports `auth = "InstancePrincipal"` and obviates the Customer Secret Key flow entirely — but D-02 locked S3-compat, so the planner has a choice to make: re-litigate D-02 to switch to the native backend (recommended given the 1.11 regression) or stay on S3-compat with a `terraform = "~> 1.10.0"` pin.

3. **GHA → OCI OIDC requires an Identity Domain + Identity Propagation Trust, not the legacy `oci_identity_identity_provider`.** Modern OCI (post-IDCS) tenancies have at least one Identity Domain (default). The Terraform resource is `oci_identity_domains_identity_propagation_trust`, and the GHA workflow exchanges the GitHub OIDC JWT for an OCI **UPST (User Principal Session Token)** via a POST to `https://<domain>/oauth2/v1/token`. This is more involved than the equivalent AWS `aws-actions/configure-aws-credentials` step — there is no official `oracle-actions/configure-oci-credentials@v1`. The community pattern (Oracle A-Team blog) is a custom shell step that does the token exchange. **The simpler alternative** — and what the Oracle official tutorial actually documents — is **OCI Session Token-based auth where a pre-generated session token is stored as a GitHub Secret** (rotated periodically). This is a small step back from pure OIDC but ships immediately and matches D-04's spirit (no long-lived API keys) since UPSTs are time-limited.

4. **Reserved Public IP cannot be attached to an instance during creation.** OCI's TF provider only allows `private_ip_id` to be set on `oci_core_public_ip` AFTER the instance exists (because the instance needs its primary VNIC and its private IP first, and those are emitted as attributes only after creation). The TF flow is: create instance with `assign_public_ip = false` → query the VNIC → get the primary private IP → assign the reserved public IP via `oci_core_public_ip.lifetime = "RESERVED"` + `private_ip_id = <data source lookup>`. Three resources, in this dependency order.

**Primary recommendation:** Stay on the S3-compat backend (CONTEXT D-02 locked, plan accepts the constraint), but **pin Terraform to `~> 1.10.0`** (NOT `>= 1.10` open-ended) until 1.11.3+ is verified against OCI. Use `null_resource` + `local-exec` for CORS (so a single `terraform apply` produces a fully-CORS-configured bucket). Use Identity Propagation Trust with impersonation onto a dedicated `gha-deployer` Service User; document the UPST-exchange shell step inline in the workflow until an official OCI GHA action exists. Wire Reserved Public IP as a separate resource with explicit `depends_on` on the instance. Delete `infra/setup.sh` per D-07; cloud-init absorbs every line — the iptables `-I INPUT 6` lines go in `runcmd:`, AFTER `package_update`/`package_upgrade` (cloud-init module order: write_files → set_timezone → groups → users → packages → runcmd, so runcmd runs LAST, which means iptables-persistent is already installed by the time we insert rules).

### Pattern 1: `backend "s3"` with OCI S3-compat + native locking (1.10.x)

**What:** Pin Terraform to 1.10.x to use `use_lockfile = true` without DynamoDB. Use OCI's S3-compat endpoint with Customer Secret Key.

**Verified-working block** [CITED: docs.oracle.com/en/learn/terraform-tfstate-file-locking-s3-backend, dev.to/drewmullen/terraform-state-with-native-s3-locking-1fck]:

```hcl
# infra/terraform/backend.tf
terraform {
  required_version = "~> 1.10.0"   # NOT >= 1.10 — avoid 1.11.2 OCI bug

  backend "s3" {
    bucket = "timeline-tfstate"
    key    = "infra/terraform.tfstate"
    region = "us-sanjose-1"          # NOTE: must match the bucket's home region

    endpoints = {
      s3 = "https://<NAMESPACE>.compat.objectstorage.us-sanjose-1.oraclecloud.com"
    }

    use_lockfile                = true   # TF 1.10+ native S3 conditional-write locking
    use_path_style              = true   # OCI S3-compat requires path-style URLs
    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_region_validation      = true
    skip_requesting_account_id  = true   # avoids an AWS STS call
    skip_s3_checksum            = true   # avoids x-amz-content-sha256 issues on OCI
  }
}
```

**Note on `force_path_style` vs `use_path_style`:** Newer S3 backends use `use_path_style`; the older `force_path_style` is deprecated. On TF 1.10+, use `use_path_style`. [CITED: developer.hashicorp.com/terraform/language/backend/s3]

**Auth env vars** (set in GHA workflow via repo Secrets):
```bash
AWS_ACCESS_KEY_ID="<oci-customer-secret-key-access-id>"
AWS_SECRET_ACCESS_KEY="<oci-customer-secret-key-secret>"
# Required for AWS CLI v2.23.5+ against OCI S3:
AWS_REQUEST_CHECKSUM_CALCULATION=when_required
AWS_RESPONSE_CHECKSUM_VALIDATION=when_required
```

**Initialization (one-time per workspace):**
```bash
cd infra/terraform
terraform init   # automatically uses backend.tf
```

### Pattern 2: `provider "oci"` with API Key OR Instance Principal OR UPST

**Three auth modes** the planner needs to pick between for the GHA execution context:

```hcl
# infra/terraform/providers.tf
terraform {
  required_providers {
    oci = {
      source  = "oracle/oci"
      version = "~> 6.0"
    }
  }
}

# Option A — API Key (laptop dev path):
provider "oci" {
  tenancy_ocid     = var.tenancy_ocid
  user_ocid        = var.user_ocid
  fingerprint      = var.fingerprint
  private_key_path = var.private_key_path
  region           = var.region
}

# Option B — Session Token (UPST) for GHA (recommended for D-04 OIDC path):
provider "oci" {
  auth                = "SecurityToken"
  config_file_profile = "DEFAULT"      # OCI CLI config + session token written by the auth step
  region              = var.region
}

# Option C — Instance Principal (if TF runs on an OCI VM, not on GHA):
provider "oci" {
  auth   = "InstancePrincipal"
  region = var.region
}
```

[CITED: registry.terraform.io/providers/oracle/oci/latest/docs — `auth` argument]. The GHA workflow writes the session token + config file to disk before `terraform init`, then `auth = "SecurityToken"` picks it up.

### Pattern 3: `oci_core_instance` for Ampere A1.Flex with cloud-init

[CITED: github.com/stealthybox/tf-oci-arm/blob/main/compute.tf, github.com/AmpereComputing/terraform-oci-ampere-a1/blob/main/ubuntu2204.tf]

```hcl
# infra/terraform/main.tf
data "oci_identity_availability_domains" "ads" {
  compartment_id = var.tenancy_ocid
}

data "oci_core_images" "ubuntu_22_04_aarch64" {
  compartment_id           = var.tenancy_ocid
  operating_system         = "Canonical Ubuntu"
  operating_system_version = "22.04"
  shape                    = "VM.Standard.A1.Flex"

  # Belt-and-braces filter for ARM image variant
  filter {
    name   = "display_name"
    values = ["^.*-aarch64-.*$"]
    regex  = true
  }
}

# infra/terraform/cloud-init.tf
locals {
  cloud_init_rendered = templatefile("${path.module}/../cloud-init.yaml", {
    repo_url = var.repo_url   # https://github.com/usbryanchlam/timeline-revamp.git
  })
}

# infra/terraform/compute.tf
resource "oci_core_instance" "timeline" {
  display_name        = "timeline-revamp-vm"
  compartment_id      = var.compartment_ocid
  availability_domain = data.oci_identity_availability_domains.ads.availability_domains[0].name
  shape               = "VM.Standard.A1.Flex"

  shape_config {
    ocpus         = 4
    memory_in_gbs = 24
  }

  source_details {
    source_type             = "image"
    source_id               = data.oci_core_images.ubuntu_22_04_aarch64.images[0].id
    boot_volume_size_in_gbs = 100
  }

  create_vnic_details {
    subnet_id        = oci_core_subnet.public.id
    assign_public_ip = false      # we attach a RESERVED IP separately
    display_name     = "timeline-vnic"
  }

  metadata = {
    ssh_authorized_keys = var.ssh_public_key
    user_data           = base64encode(local.cloud_init_rendered)
  }
}

# Reserved public IP — attached AFTER instance creation because TF cannot
# resolve the private_ip_id until the VNIC exists.
data "oci_core_vnic_attachments" "primary" {
  compartment_id = var.compartment_ocid
  instance_id    = oci_core_instance.timeline.id
}

data "oci_core_private_ips" "primary" {
  vnic_id = data.oci_core_vnic_attachments.primary.vnic_attachments[0].vnic_id
}

resource "oci_core_public_ip" "timeline" {
  compartment_id = var.compartment_ocid
  display_name   = "timeline-reserved-ip"
  lifetime       = "RESERVED"
  private_ip_id  = data.oci_core_private_ips.primary.private_ips[0].id
}
```

**Critical:** The `private_ip_id` argument is what binds the reserved IP to the instance's primary VNIC. Setting `lifetime = "RESERVED"` without `private_ip_id` creates an unattached reserved IP (still valid, but useless). The data-source-via-instance pattern is the documented bridge between the two resources [CITED: docs.oracle.com/en/learn/oci-attach-reserved-ip].

### Pattern 4: `oci_objectstorage_bucket` + CORS via `null_resource`

**What:** The OCI TF provider does NOT support CORS rules natively (verified — argument list does not include `cors_rules`). Apply CORS via the AWS-S3-compat endpoint inside a `null_resource` that runs after bucket creation.

```hcl
# infra/terraform/storage.tf
data "oci_objectstorage_namespace" "this" {
  compartment_id = var.tenancy_ocid
}

resource "oci_objectstorage_bucket" "photos" {
  compartment_id = var.compartment_ocid
  namespace      = data.oci_objectstorage_namespace.this.namespace
  name           = "timeline-photos"
  access_type    = "NoPublicAccess"     # PARs are minted per-object; no public-list access

  versioning = "Enabled"

  # Optional — defined_tags for cost tracking
  freeform_tags = {
    "managed-by"  = "terraform"
    "phase"       = "08.1"
    "environment" = "production"
  }
}

resource "null_resource" "photos_cors" {
  # Re-apply when CORS rules change
  triggers = {
    cors_rules_hash = sha256(jsonencode(var.photos_cors_rules))
    bucket_id       = oci_objectstorage_bucket.photos.id
  }

  provisioner "local-exec" {
    command = <<-EOT
      export AWS_REQUEST_CHECKSUM_CALCULATION=when_required
      export AWS_RESPONSE_CHECKSUM_VALIDATION=when_required
      aws s3api put-bucket-cors \
        --endpoint-url "https://${data.oci_objectstorage_namespace.this.namespace}.compat.objectstorage.${var.region}.oraclecloud.com" \
        --bucket "${oci_objectstorage_bucket.photos.name}" \
        --cors-configuration '${jsonencode({ CORSRules = var.photos_cors_rules })}'
    EOT
  }

  depends_on = [oci_objectstorage_bucket.photos]
}
```

`var.photos_cors_rules` shape:
```hcl
variable "photos_cors_rules" {
  type = list(object({
    AllowedOrigins = list(string)
    AllowedMethods = list(string)
    AllowedHeaders = list(string)
    ExposeHeaders  = list(string)
    MaxAgeSeconds  = number
  }))
  default = [{
    AllowedOrigins = ["https://timeline.bryanlam.dev", "http://localhost:5173"]
    AllowedMethods = ["GET", "HEAD", "PUT"]
    AllowedHeaders = ["*"]
    ExposeHeaders  = ["ETag", "Content-Length"]
    MaxAgeSeconds  = 3600
  }]
}
```

**Important:** GHA `ubuntu-latest` runners pre-install AWS CLI v2 + `aws-sdk-v2` defaults; `AWS_REQUEST_CHECKSUM_CALCULATION` is the load-bearing env var [CITED: project memory `feedback_oci_cors_via_s3.md`]. The same access key + secret used for the state backend works for the CORS call.

### Pattern 7: GHA workflow shape (plan + apply + drift)

```yaml
# .github/workflows/terraform.yml
name: terraform
on:
  pull_request:
    paths: ['infra/terraform/**', '.github/workflows/terraform.yml']
  push:
    branches: [main]
    paths: ['infra/terraform/**', '.github/workflows/terraform.yml']
  schedule:
    - cron: '0 12 * * 1'   # Monday noon UTC — D-18

permissions:
  contents: read
  id-token: write          # REQUIRED for OIDC token issuance
  issues: write            # for drift Issue creation
  pull-requests: write     # for plan PR comments

env:
  TF_VERSION: '1.10.7'     # planner pins exact version at plan time
  AWS_ACCESS_KEY_ID: ${{ secrets.OCI_S3_ACCESS_KEY }}
  AWS_SECRET_ACCESS_KEY: ${{ secrets.OCI_S3_SECRET_KEY }}
  AWS_REQUEST_CHECKSUM_CALCULATION: when_required
  AWS_RESPONSE_CHECKSUM_VALIDATION: when_required

jobs:
  plan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Authenticate to OCI via OIDC token exchange
        run: |
          # GitHub provides ACTIONS_ID_TOKEN_REQUEST_URL and ACTIONS_ID_TOKEN_REQUEST_TOKEN
          GH_OIDC=$(curl -sH "Authorization: bearer $ACTIONS_ID_TOKEN_REQUEST_TOKEN" \
                    "$ACTIONS_ID_TOKEN_REQUEST_URL&audience=https://oraclecloud.com" \
                    | jq -r '.value')
          # Exchange for an OCI UPST:
          UPST=$(curl -sX POST "https://${{ vars.OCI_DOMAIN_URL }}/oauth2/v1/token" \
                  -H 'Content-Type: application/x-www-form-urlencoded' \
                  --data-urlencode "grant_type=urn:ietf:params:oauth:grant-type:token-exchange" \
                  --data-urlencode "requested_token_type=urn:oci:token-type:upst" \
                  --data-urlencode "subject_token=$GH_OIDC" \
                  --data-urlencode "subject_token_type=urn:ietf:params:oauth:token-type:jwt" \
                  | jq -r '.access_token')
          # Write OCI CLI config + token to disk for auth = "SecurityToken":
          mkdir -p ~/.oci
          echo "$UPST" > ~/.oci/upst.token
          cat > ~/.oci/config <<EOF
          [DEFAULT]
          region=${{ vars.OCI_REGION }}
          tenancy=${{ vars.OCI_TENANCY_OCID }}
          security_token_file=~/.oci/upst.token

        working-directory: infra/terraform

          terraform fmt -check
          terraform init -input=false
          terraform validate

      - name: Terraform plan
        id: plan
        working-directory: infra/terraform
        run: terraform plan -no-color -out=tfplan -detailed-exitcode
        continue-on-error: true   # exit code 2 = changes; we want to surface, not fail

      - name: Comment plan on PR
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const output = `### Terraform plan\n\n\`\`\`\n${{ steps.plan.outputs.stdout }}\n\`\`\``;
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner, repo: context.repo.repo,
              body: output,
            })

      - name: Save plan artifact
        if: github.event_name == 'push' && github.ref == 'refs/heads/main'
        uses: actions/upload-artifact@v4
        with:
          name: tfplan
          path: infra/terraform/tfplan

      - name: Open drift issue
        if: github.event_name == 'schedule' && steps.plan.outputs.exitcode == '2'
        uses: actions/github-script@v7
        with:
          script: |
            const title = '[drift] Terraform plan detected unexpected changes';
            const body = `\`\`\`\n${{ steps.plan.outputs.stdout }}\n\`\`\``;

            const existing = await github.rest.issues.listForRepo({
              owner: context.repo.owner, repo: context.repo.repo,
              labels: 'drift,infrastructure', state: 'open'
            });

            if (match) {
              await github.rest.issues.createComment({
                issue_number: match.number,
                owner: context.repo.owner, repo: context.repo.repo,
                body: `New drift detection at ${new Date().toISOString()}:\n${body}`
              });
            } else {
              await github.rest.issues.create({
                owner: context.repo.owner, repo: context.repo.repo,
                title, body, labels: ['drift', 'infrastructure']
              });
            }

  apply:
    needs: plan
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest

      group: terraform-apply
      cancel-in-progress: false
    steps:
      - uses: actions/checkout@v4
      - name: Authenticate to OCI via OIDC token exchange

          # ... same as plan job ...
      - uses: hashicorp/setup-terraform@v3

          terraform_version: ${{ env.TF_VERSION }}
      - uses: actions/download-artifact@v4
        with:
          name: tfplan
          path: infra/terraform/
      - name: Terraform apply
        working-directory: infra/terraform
        run: |
          terraform init -input=false
          terraform apply -input=false -no-color tfplan
```

### Anti-Patterns to Avoid

- **`force_path_style = true`** — DEPRECATED in newer S3 backends. Use `use_path_style = true` instead. Mixing both may cause silent fallbacks. [CITED: TF S3 backend changelog]
- **Setting `private_ip_id = ""` on `oci_core_public_ip`** — The Oracle tutorial does this and then attaches via cloud-init `oci network public-ip update`. This works but splits the source of truth between TF and a shell script. **Use the data-source bridge pattern** (`data.oci_core_private_ips`) to keep TF as the single source of truth.
- **Skipping `depends_on` on the CORS `null_resource`** — Without explicit `depends_on = [oci_objectstorage_bucket.photos]`, TF may run the `local-exec` in parallel with bucket creation and the `aws s3api` call will fail with a NotFound error.
- **Storing Customer Secret Keys in `terraform.tfvars`** — They belong in GHA Secrets only. Use `*.tfvars.example` for shape documentation; never commit real secrets.
- **Using `oci_identity_identity_provider` for OIDC** — That resource is for SAML federation only. OIDC requires Identity Domains + `oci_identity_domains_identity_propagation_trust`.
- **Cross-region state bucket** — The state bucket region MUST match the OCI region you're deploying resources into for S3-compat path style to resolve correctly. CONTEXT D-11 says "user's home region"; planner should default to that.
- **Hardcoding `availability_domains[0]`** — If AD-1 is out of A1 capacity (Pitfall 1 below), TF will fail. Consider a `count = length(... .availability_domains)` retry pattern OR document the AD index as a variable so the operator can flip it without a code edit.
- **Building the state bucket via TF** — Chicken-and-egg: the bucket is the state backend. Bootstrap via OCI CLI (D-09). The bucket then becomes a "managed-outside-of-TF" resource — that's fine and intentional.

## Common Pitfalls

### Pitfall 1: Ampere A1 free-tier "Out of host capacity"
**What goes wrong:** `terraform apply` fails with `Error: 500-InternalError, Out of host capacity` when creating the A1.Flex instance. OCI free-tier A1 capacity is famously contested; the resource may be unavailable for days in popular regions.
**Why it happens:** OCI has a finite pool of always-free A1 hosts per AZ; demand exceeds supply in many regions.
**How to avoid:** (a) Try a different AD (`availability_domains[1]` or `[2]`); (b) try at off-peak hours; (c) reduce shape to 2 OCPU / 12 GB which sometimes has more capacity; (d) accept that the first apply may need 5-20 retries — TF will recreate the plan cleanly on each `terraform apply`. **Document this in DEPLOY.md** as a known constraint, not a TF bug. [CITED: hitrov/oci-arm-host-capacity GitHub repo, OCI Cloud Customer Connect forums]
**Warning signs:** Plan succeeds, apply fails immediately on `oci_core_instance.timeline` with HTTP 500.

### Pitfall 2: TF 1.11.2 breaks `use_lockfile` on OCI buckets
**What goes wrong:** Upgrading Terraform to 1.11.x causes every plan/apply to fail with `InvalidArgument: x-amz-content-sha256 must be UNSIGNED-PAYLOAD or a valid sha256 value`.
**Why it happens:** TF 1.11.2's S3 backend started sending `x-amz-content-sha256: STREAMING-UNSIGNED-PAYLOAD-TRAILER` (or similar) headers that OCI's S3-compat endpoint rejects. [VERIFIED: hashicorp/terraform#36742]
**How to avoid:** Pessimistic pin `required_version = "~> 1.10.0"` (NOT `>= 1.10`). Re-test on 1.11.3+ before pinning forward.
**Warning signs:** A working stack stops working after a TF version bump; error mentions `x-amz-content-sha256`.

### Pitfall 3: cloud-init `runcmd` order vs. iptables-persistent install
**What goes wrong:** A naive cloud-init that puts `apt-get install -y iptables-persistent` in `runcmd:` followed by `iptables -I INPUT 6 ...` may fire the iptables commands before the package is installed if the operator confuses cloud-init's module order.
**Why it happens:** cloud-init's documented order is `write_files → set_timezone → groups → users → packages → runcmd`. As long as `iptables-persistent` is in the top-level `packages:` block (NOT in runcmd), this works. But putting `apt-get install` INSIDE runcmd alongside the iptables commands creates a race.
**How to avoid:** Use the top-level `packages:` array for apt installs. Reserve `runcmd:` for commands that need orchestration order. [VERIFIED: docs.cloud-init.io reference]
**Warning signs:** `cloud-init status --long` shows `runcmd` errors mentioning "iptables-save: command not found" or "netfilter-persistent: command not found".

### Pitfall 4: Cloud-init runs ONCE — re-applying TF doesn't re-bootstrap
**What goes wrong:** Operator edits `infra/cloud-init.yaml`, runs `terraform apply`. TF computes a `metadata.user_data` change on `oci_core_instance.timeline`. The provider's behavior here is to either (a) require instance replacement (destroying the VM and its boot volume, losing `pgdata`) or (b) silently update the metadata but leave the running VM untouched.
**Why it happens:** `metadata` changes on `oci_core_instance` are an "in-place" update per the provider docs, BUT cloud-init only runs on first boot. So the metadata is updated but never executed.
**How to avoid:** Document this: cloud-init changes only take effect on instance replacement. Use Compose + git pull for runtime app changes; use TF + instance replacement for cloud-init changes. Phase 9 should add a `null_resource` with `remote-exec` for ad-hoc post-boot config if needed.
**Warning signs:** Operator edits cloud-init, sees TF report "in-place update", expects new config to be applied, sees no change on the VM.

### Pitfall 5: Reserved Public IP attach race
**What goes wrong:** TF runs in parallel — the `oci_core_public_ip` resource may try to read `data.oci_core_private_ips` BEFORE the instance's VNIC is fully attached, causing `private_ips[0]` to be empty.
**Why it happens:** OCI's VNIC attachment is asynchronous; the instance can be "RUNNING" before the primary VNIC's private IP is queryable.
**How to avoid:** Add explicit `depends_on = [oci_core_instance.timeline]` on the data sources, AND consider a small `time_sleep` resource (30s) between instance creation and IP attachment to give OCI's eventual-consistency layer time to settle. [VERIFIED: oracle/terraform-provider-oci#1565 discussion]
**Warning signs:** First apply fails with `data.oci_core_private_ips.primary.private_ips is empty`; retrying the same apply succeeds.

### Pitfall 6: Identity Propagation Trust requires an Identity Domain
**What goes wrong:** Legacy OCI tenancies (pre-IDCS migration) may not have an Identity Domain. `oci_identity_domains_*` resources fail with "Tenancy is not domain-enabled".
**Why it happens:** OCI introduced Identity Domains in 2023; older tenancies were migrated automatically but a few outliers remain.
**How to avoid:** Verify with `oci iam domain list --compartment-id <tenancy-ocid>` BEFORE writing the IAM TF. If no Default domain exists, the trust must be created via the legacy IAM endpoints (rare; document as a known unknown).
**Warning signs:** `oci_identity_domains_identity_propagation_trust` creation fails with 404 NotFound on the IDCS endpoint.

### Pitfall 7: OCID format confusion
**What goes wrong:** `var.tenancy_ocid` and `var.compartment_ocid` look similar — both start with `ocid1.` — but have different middle segments (`tenancy.oc1..` vs `compartment.oc1..`). Pasting tenancy where compartment is expected (or vice versa) causes confusing IAM policy failures.
**Why it happens:** Both are long opaque strings; easy to swap.
**How to avoid:** Use `var.compartment_ocid` everywhere except `data.oci_identity_availability_domains.compartment_id` (which can use tenancy). Document the distinction in `terraform.tfvars.example` with a one-line comment for each.
**Warning signs:** Policies "created successfully" but never grant access; instance principal calls return 404 on bucket reads.

### Pitfall 8: Default tags namespace pollution
**What goes wrong:** OCI auto-applies an `Oracle-Tags` defined-tag namespace (`CreatedBy`, `CreatedOn`) to every resource. TF shows these as "spurious" diffs on subsequent plans.
**Why it happens:** OCI's default tagging on resource creation; TF doesn't know to ignore them.
**How to avoid:** Add a `lifecycle { ignore_changes = [defined_tags["Oracle-Tags.CreatedBy"], defined_tags["Oracle-Tags.CreatedOn"]] }` block on each resource that exhibits this. Or globally suppress via a `lifecycle { ignore_changes = [defined_tags] }` if no defined tags are actually managed by TF. [VERIFIED: oracle/terraform-provider-oci discussions]
**Warning signs:** Every `terraform plan` shows ~5-10 resources with "defined_tags change" but no actual config drift.

### Pitfall 9: `aws s3api put-bucket-cors` SDK v2 checksum mismatch
**What goes wrong:** Running `aws s3api put-bucket-cors --endpoint-url <oci-s3-compat>` from a GHA runner (AWS CLI v2.23.5+) fails with `MissingContentLength` or `InvalidArgument` errors.
**Why it happens:** Newer AWS SDK v2 sends streaming checksums by default; OCI's S3-compat endpoint doesn't fully support them.
**How to avoid:** Set `AWS_REQUEST_CHECKSUM_CALCULATION=when_required` and `AWS_RESPONSE_CHECKSUM_VALIDATION=when_required` in the env before the `local-exec` block. [CITED: project memory `feedback_oci_cors_via_s3.md`]
**Warning signs:** CORS apply fails with `MissingContentLength`; downgrading AWS CLI fixes it (but the env-var workaround is cleaner).

### Pitfall 10: PR-on-fork OIDC silently doesn't fire
**What goes wrong:** External contributor opens a PR from a fork; the OIDC trust step fails because forks don't get OIDC tokens by default.
**Why it happens:** GitHub's security default — fork-originated workflows can't request OIDC tokens unless the maintainer approves them.
**How to avoid:** This is a solo project (CONTEXT D-19 deferred section flags this). Document: "external PRs would require manual maintainer approval to run TF workflows; until then, fork-PR plan jobs fall back to `terraform fmt -check` + `validate` only."
**Warning signs:** A fork-PR shows "no OIDC token available"; the workflow either fails the auth step or silently skips it depending on the workflow's `if:` guards.

## Open Questions

1. **Which TF backend approach: S3-compat (D-02 locked) or native `backend "oci"` (TF 1.12+, simpler)?**
   - What we know: D-02 locked S3-compat. But the native backend solves the 1.11.2 regression entirely and supports `auth = "InstancePrincipal"` natively.
   - What's unclear: whether D-02 is re-litigatable given the regression discovery.
   - Recommendation: Stay on S3-compat for 8.1 (D-02 binding); add a follow-up note for v2 migration to native backend once Terraform 1.12+ is stable and the team has had a successful 8.1 ship.

2. **Two-identity OIDC (D-19) vs single-identity?**
   - What we know: D-19 prefers separated identities (apply on main, read-only on PR).
   - What's unclear: whether OCI Propagation Trust supports multiple subject-mapping rules in one trust, or requires two separate trusts.
   - Recommendation: One trust with two `impersonation_service_users` rules if supported; otherwise two trusts. Either works.

3. **CORS rule extraction: does the existing Phase 6 bucket have rules to preserve?**
   - What we know: Phase 6 set CORS via `aws s3api put-bucket-cors` (per memory). Current state unknown.
   - What's unclear: whether the rules match the production domain `https://timeline.bryanlam.dev` or are still dev-only.
   - Recommendation: Planner runs `aws s3api get-bucket-cors` at plan time; codify whatever's there into `var.photos_cors_rules` default; widen to include production domain if missing.

4. **Cloud-init replacement semantics on `oci_core_instance.metadata.user_data` change?**
   - What we know: Provider docs are ambiguous about whether this forces replacement.
   - What's unclear: whether existing instances get the new metadata WITHOUT replacement (no effect since cloud-init runs once) or get replaced (data-destroying).
   - Recommendation: Test with a trivial edit (e.g., comment line) during the first execution; document the actual behavior. Either way, surface to operators that "cloud-init changes require instance replacement to take effect."

5. **Reserved IP attach timing: race condition or reliable?**
   - What we know: TF provider issue #1565 hints at flakiness when attaching reserved IP via data sources.
   - What's unclear: whether `depends_on` is sufficient, or whether `time_sleep` is needed.
   - Recommendation: Start with `depends_on` only; if the first apply fails with "private_ips empty", add a 30s `time_sleep`.

### Known Threat Patterns for the OCI + TF + GHA stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Long-lived API key in GitHub Secrets leaks | Information Disclosure | OIDC trust (D-04) — no long-lived keys except state-backend SK |
| State file contains secrets (Postgres password, OCI OCIDs) | Information Disclosure | OCI bucket default encryption + private access; D-13 `.env` stays out of TF; rotation procedure in DEPLOY.md |
| Plan-on-PR exposes private OCIDs in PR comment | Information Disclosure | Acceptable for a private repo + solo project; document; redact if repo goes public |
| Compromised PR identity escalates to apply | Elevation of Privilege | Tight OIDC subject scoping per D-19: PR identity is read-only (`inspect`) ONLY, NOT `manage` |
| `terraform apply` race / split-brain | Tampering | `use_lockfile = true` + concurrency group `terraform-apply` in GHA |
| Drift caused by unauthorized OCI Console change | Tampering | Weekly drift cron + Issue alert (D-18) |
| GHA runner compromise exfiltrates UPST | Spoofing | UPST is short-lived; rotated per workflow run; no persistent storage |
| Cloud-init YAML injection via `templatefile()` vars | Tampering | Vars are operator-controlled (compartment OCID, repo URL); not user input. Safe. |
| Fork-PR with malicious TF tries to run apply | Spoofing / Elevation | Fork PRs cannot get OIDC tokens without maintainer approval; default GHA security |
| Customer SK compromised → state-bucket tampering | Tampering | Rotate SK quarterly (document in DEPLOY.md § "Operations"); enable bucket versioning to recover |
