### Gap 3 — cloud-init.yaml has 4 known bugs (surfaced 2026-05-30 during Phase 8 Wave 3)
**What's missing:** `infra/cloud-init.yaml` does not produce a fully-working VM on first boot. Operator must apply 4 in-place fixes after SSHing in. Captured here so a re-provision today won't repeat the surprise.

**The 4 bugs:**
1. **`docker-compose-plugin` package install fails** — package isn't in Ubuntu 22.04 jammy default apt repos. cloud-init's `packages:` directive can't pull it. Fix: either add Docker's official apt repo via `apt:` config and re-add `docker-compose-plugin`, OR switch to `docker-compose-v2` (the Ubuntu universe package — may or may not exist on jammy; verify before relying on it).
2. **iptables `ACCEPT 80/443` rules appended after the catch-all `REJECT`** — `iptables -A INPUT` appends to the end of the chain. Need `iptables -I INPUT 5` (insert at position 5, before the REJECT). cloud-init's `runcmd` step that adds 80/443 needs to use `-I` not `-A`.
3. **`git clone` runs even when `package_update_upgrade_install` errored** — cloud-init's runcmd block isn't conditional on prior-step success by default. Either gate the clone behind a success-marker file, or detect partial install and bail before proceeding.
4. **`/var/cache/nginx/public_reel` directory missing** — `ops/nginx/timeline.conf` declares `proxy_cache_path /var/cache/nginx/public_reel ...` but cloud-init doesn't pre-create it. nginx -t fails until operator runs `mkdir + chown -R www-data:www-data`. Fix: add a `runcmd` step that creates the cache dir with `www-data:www-data` ownership BEFORE nginx is enabled.

**Additional findings (not strictly cloud-init bugs but related v1 deploy gaps):**

5. **OCI PEM file UID mismatch.** The Dockerfile runs the container as UID 1001; DEPLOY.md instructs the operator to SCP the PEM owned by `ubuntu` (UID 1000) with mode 600. UID 1001 can't read it → `EACCES` on first photo upload. Fix: add `sudo chown 1001:1001 /opt/timeline-revamp/.oci/timeline-revamp.pem` to the Post-Provision SCP block in `infra/DEPLOY.md`. Longer-term fix is to switch `server/oci/parClient.ts` to Instance Principal auth (IAM resources for it already exist from 08.1-02) and eliminate the PEM-in-container entirely.

6. **Bucket access_type design correction.** `storage.tf` originally set `access_type = "NoPublicAccess"` on the theory that PARs would be minted per-object for both reads and writes. The actual code only mints write PARs; `getMasterBuffer` does an unauthenticated fetch of the public URL for thumbnail generation, and `<img src>` tags in the SPA render the public URL directly. Fixed in commit `de2a855` by switching to `access_type = "ObjectRead"`. Future Path B hardening would mint short-TTL read PARs server-side and pass PAR URLs to the client.

**Severity:** Medium. Cloud-init runs once per VM lifecycle; manual fixes work, but the IaC promise of "reproducible via `terraform apply`" is broken until these are patched.
**Resolution path:** Phase 8.1.1 (or as early scope of Phase 9). Patches to `infra/cloud-init.yaml` + `infra/terraform/storage.tf` (already done for bucket access) + `infra/DEPLOY.md` (PEM chown step). Verify by `terraform taint oci_core_instance.timeline + terraform apply` then walk through Phase 8 Wave 3 deploy from scratch.
