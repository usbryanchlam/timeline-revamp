<objective>
Add the authenticated-tree chrome: a fixed bottom navigation bar (Reel | Trips | Me) and a route guard component that wraps `/app/*`. The guard is a stub today (pass-through) but is the structural seam Phase 4 will use to drop in Auth0.

Purpose: Satisfy APP-01 ("Bottom nav renders on authenticated routes only") and the second half of APP-03 ("private tree wrapped"). The nav lives only inside `AppLayout`, so by construction it never appears on the public reel or per-handle pages — APP-01 is enforced by route-tree topology, not by a runtime check.

The guard is a small but high-leverage piece of structure: by introducing `<RequireAuth>` as a separate component now (returning `{children}`), Phase 4's auth work becomes a body-only edit to `RequireAuth.tsx` — no churn in route components, no churn in `AppLayout`.

**Z-index / layout collision to handle:** `/app/` (the index route) renders the same Reel as `/`. The Reel is `.reel-root` (`src/index.css:48-60`: `height: 100dvh; overflow: hidden`), and the ChapterRail inside it is anchored at `bottom: max(env(safe-area-inset-bottom), 32px)` (`src/reel/ChapterRail.tsx:34`). On `/app/`, the new BottomNav sits at the same viewport bottom (64px tall, z-40). Without intervention the BottomNav covers the ChapterRail. Solution: only on `/app/`, lift the rail above the BottomNav by adding extra bottom padding to the reel surface. We do NOT hide the BottomNav on `/app/` — that would contradict APP-01's verbatim spec ("renders on /app/* only"; the index `/app/` IS an `/app/*` route).

Output: `BottomNav` and `RequireAuth` components, `AppLayout` wired to use both, the bottom nav functional and styled per DESIGN.md tokens, and the rail/nav collision resolved on `/app/`.
</objective>

  **Why `pb-16`:** the BottomNav is fixed-positioned with `h-16` (64px). Without bottom padding on the scrollable content area, the last items in Trips/Me would slide under the nav. 16 × 4 = 64px matches exactly. Trips/Me are normal scroll content, so this padding is enough for them.

  **Imports:** `Outlet` from `react-router`, `RequireAuth` from `@/components/RequireAuth`, `BottomNav` from `@/components/BottomNav`.

  Keep the Phase 4 forward-compat comment block from 03-01 — it still applies.

  **Sub-change B — Resolve the Reel collision in `src/routes/AppReelRoute.tsx`.** The Reel is `.reel-root` with `height: 100dvh; overflow: hidden` (`src/index.css:48-60`), and the ChapterRail is anchored to the viewport bottom at `bottom: max(env(safe-area-inset-bottom), 32px)` (`src/reel/ChapterRail.tsx:34`). The `pb-16` from sub-change A does NOT help here because `.reel-root` is `100dvh` and `overflow: hidden` — the ChapterRail's `bottom: …` is computed against the reel surface, which still extends to viewport bottom, putting the rail directly under the BottomNav.

  Fix: wrap the rendered Reel inside `AppReelRoute` with a host class (`.app-reel-host`) that the CSS uses to lift the ChapterRail. Concretely:

  ```tsx
  // src/routes/AppReelRoute.tsx
  return (
    <div className="app-reel-host">
      {reduced ? <ReducedMotionReel /> : <Reel />}
    </div>
  );
  ```

  **Sub-change C — Add the lifting CSS rule to `src/index.css`** (place it adjacent to the existing `.reel-root` rule at `src/index.css:48-60`):

  ```css
  /* When the reel renders inside AppLayout, the BottomNav (fixed, h-16 = 64px)
     would otherwise cover the ChapterRail (anchored at bottom). Push the rail
     above the nav by reserving its height + safe-area padding. */
  .app-reel-host .chapter-rail,
  .app-reel-host [data-chapter-rail] {
    bottom: calc(4rem + max(env(safe-area-inset-bottom), 32px));
  }
  ```

  Note: this assumes the ChapterRail root has either a `chapter-rail` class or a `data-chapter-rail` attribute. Check `src/reel/ChapterRail.tsx` — if neither exists, add `data-chapter-rail` to the rail's outer element in that file (one-line change, no behavior impact). Pick whichever lands cleaner.

  **Why this approach over the alternative:** Hiding BottomNav on `/app/` would technically resolve the collision but contradicts APP-01's verbatim "renders on /app/* only" — `/app/` IS an `/app/*` route. Lifting the rail keeps APP-01 satisfied verbatim and preserves the visual rhythm DESIGN.md describes (rail visible, nav visible, both legible).</action>
