Architectural note (corrects an earlier inversion in the draft of this plan): 08-RESEARCH §"Architecture Patterns" Pattern 1 and §"Code Example 2" specify that the Vite `dist/` is baked into the API container image AND served by Hono via `hono/bun`'s `serveStatic`.     New block:
    ```nginx
    # -----------------------------------------------------------------------
    # Vite static assets -- hash-fingerprinted filenames (index-abc123.js) so
    # a one-year immutable cache is safe. Reduces repeat-visit bandwidth to
    # near-zero (RESEARCH Open Question 4 / Pitfall 9). Phase 7's
    # location / fallback caught these requests with no cache header.
    #
    # IMPORTANT: per RESEARCH §"Architecture Patterns" Pattern 1, the Vite
    # dist/ lives INSIDE the API container (Dockerfile runtime stage COPYs
    # from the builder stage). The API serves /assets/* via Hono `serveStatic`
    # mounted in server/index.ts (08-01 Task 1). Nginx is host-installed and
    # has NO filesystem path to /app/dist/, so we proxy_pass here and only
    # overlay cache headers on the response. A `try_files` directive against
    # a non-existent host path would 404 every asset request.
    # -----------------------------------------------------------------------
    location ~ ^/assets/ {
        proxy_pass http://127.0.0.1:8787;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        expires 1y;
        add_header Cache-Control "public, immutable" always;
    }
    ```

    Two notes on the directive choices:
    - `expires 1y;` and `add_header Cache-Control "public, immutable" always;` together produce both the Expires header (~365 days out) and a Cache-Control with public + immutable. The `always` modifier ensures the header appears on 4xx/5xx responses too (Nginx default elides custom add_header on non-2xx).
    - We do NOT add `proxy_cache` to this block: the API's response is already deterministic per `$uri` (the hash filename is in the URL), and Nginx-side caching layered on top would just duplicate the Hono read with no win. Keep this block stateless; the long Cache-Control header instructs the BROWSER and any intermediate CDN to cache.

    Do NOT add any TLS / listen 443 directives in this edit. certbot --nginx injects those during Task 2's interactive run, and Task 3 commits the result back to the repo.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public internet -> Nginx :80 | HTTP requests; only the certbot HTTP-01 challenge path and the 301 redirect to HTTPS run here. |
| Public internet -> Nginx :443 | TLS-terminated; Nginx is the trust anchor for the cert chain. Cipher suites + TLS versions follow certbot's `options-ssl-nginx.conf` (Mozilla intermediate profile by default). |
| Nginx :443 -> API loopback 127.0.0.1:8787 | Same host; cleartext over loopback is safe (no network observer). Carries both /api/* and /assets/* (the latter is served by Hono `serveStatic` from the API container's baked dist/). |
| Nginx file system -> /etc/letsencrypt/live/.../privkey.pem | privkey.pem is mode 0600 owned by root (certbot default); only the Nginx worker (root master, www-data workers) reads it. |
| Repo working tree -> /etc/nginx/conf.d via symlink | The symlink is owned by root; the target is owned by ubuntu. `nginx -t` runs in root context and dereferences the symlink. Bad commits get caught by `nginx -t` before reload. |
