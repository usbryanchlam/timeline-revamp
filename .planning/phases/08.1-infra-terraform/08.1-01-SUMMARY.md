## Exact `terraform init` Command

Operator runs the following after the Bootstrap section is complete (anonymized — substitute tenant-specific values):

```bash
cd infra/terraform
export AWS_ACCESS_KEY_ID="<OCI_S3_ACCESS_KEY>"
export AWS_SECRET_ACCESS_KEY="<OCI_S3_SECRET_KEY>"
export AWS_REQUEST_CHECKSUM_CALCULATION=when_required
export AWS_RESPONSE_CHECKSUM_VALIDATION=when_required

terraform init \
  -backend-config="region=<REGION>" \
  -backend-config="endpoints={s3=\"https://<NAMESPACE>.compat.objectstorage.<REGION>.oraclecloud.com\"}"
```

Where:
- `<REGION>` = OCI home region (e.g., `us-sanjose-1`)
- `<NAMESPACE>` = `oci os ns get --query data --raw-output` (tenant-specific Object Storage namespace)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Outputs.tf creation split between Task 1 (placeholder) and Task 2 (real declarations)**
- **Found during:** Task 1 (Scaffold Terraform stack)
- **Issue:** Plan Task 1 specifies that `outputs.tf` is created with `output "public_ip" { value = oci_core_public_ip.timeline.ip_address }` AND that Task 1's verify suite must pass `terraform validate`. These two requirements are mutually exclusive: `oci_core_public_ip.timeline` is declared in `compute.tf` which is not created until Task 2. Declaring outputs that reference yet-to-exist resources causes `terraform validate` to fail with "Reference to undeclared resource".
- **Fix:** Wrote `outputs.tf` in Task 1 as a header-comment-only placeholder that explicitly documents the deferral. In Task 2, expanded `outputs.tf` to declare `public_ip` and `instance_ocid` alongside the `compute.tf` resources they reference. Both commits pass `terraform fmt -check` AND `terraform validate`. File appears in Task 1's `files_modified` list (matches plan scope) AND in Task 2's diff (matches verify acceptance).
- **Files modified:** `infra/terraform/outputs.tf` (Task 1: stub; Task 2: real outputs)
- **Verification:** `terraform validate` succeeds after each of the three commits independently.
- **Committed in:** `9b5d80c` (stub) + `e4bcbdc` (real)

## Issues Encountered

- **Shell `grep` is aliased to `ugrep` with `-G` mode** in this environment, which interprets `${...}` patterns differently from BSD/GNU grep. Verify-suite greps that include `${path.module}` returned spurious mismatches under the alias; switched to `command grep` for the verify automation. The underlying file content is correct — only the alias was misbehaving. (Project memory: `feedback_grep_guard_vs_comments.md` — comment-vs-code grep brittleness; this is a related but distinct alias-vs-binary issue. Worth flagging for future executors.)
- **Worktree base was on the pre-correction commit** (`3e16f83`) instead of the expected `e9ae81a` (which corrected the `photos_cors_rules` default to include `PUT` and `x-amz-version-id`). Hard-reset to expected base before execution; the photos_cors_rules variable in `variables.tf` reflects the corrected defaults (AllowedMethods: GET/HEAD/PUT; ExposeHeaders: ETag/Content-Length/x-amz-version-id).
