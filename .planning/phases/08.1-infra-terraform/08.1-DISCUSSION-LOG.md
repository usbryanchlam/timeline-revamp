## Area 4: GHA apply gate + drift detection

### Q1: How should `terraform apply` be triggered from GitHub Actions?

| Option | Description | Selected |
|--------|-------------|----------|
| Manual approval via GitHub Environments (Recommended) | `production` Environment with required reviewer; pause-and-approve in Actions UI | ✓ |
| Auto-apply on push to main (with concurrency lock) | Fast iteration; no confirmation step | |

**User's choice:** Manual approval via GitHub Environments.
**Notes:** Reviewer = user (solo project). Apply job pauses in Actions UI until approved.

### Q2: Should there be scheduled drift detection?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — weekly cron, GH issue on drift (Recommended) | `terraform plan -detailed-exitcode` Monday noon UTC | ✓ |

| No — skip for v1 | One less workflow to maintain | |

**User's choice:** Weekly cron with GH issue.
**Notes:** Idempotent — appends to an existing open drift issue rather than creating duplicates.

### Q3: How should the OIDC trust policy be scoped?

| Option | Description | Selected |
|--------|-------------|----------|
| Tight: only `repo:usbryanchlam/timeline-revamp:ref:refs/heads/main` (Recommended) | Apply identity scoped to main; PR identity TBD by planner | ✓ |

**Notes:** PR-side identity decided at plan-phase time. Default to a separate read-only identity for plan visibility on PRs unless OIDC subject syntax makes it impractical.
