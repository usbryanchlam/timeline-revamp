<objective>
Wire Auth0 end-to-end. Frontend uses Auth0's React SDK, scoped to the `/app/*` tree only (so the public reel stays SDK-free per AUTH-04). Backend validates JWTs against Auth0's JWKS endpoint via the `jose` library, lazily provisions a `users` row on first authenticated call (AUTH-03), and exposes `/api/me` + `/api/me/handle` so a fresh user can pick a handle on first visit.

This plan also delivers handle reservation (AUTH-05/06/07) inside the picker modal. ROADMAP.md Phase 4's plan body explicitly says "handle picker UI" in the 04-02 line, which is impossible to deliver without AUTH-05 (the regex), AUTH-06 (the reserved-word list), and AUTH-07 (the prompt-on-first-visit behavior). REQUIREMENTS.md's traceability table currently maps AUTH-05..07 to Phase 7 (W7); Task 7 of this plan reconciles that drift by editing REQUIREMENTS.md to point at Phase 4. Phase 7 (PUBLIC URLs) still owns the *consumption* of handles (`/u/:handle` rendering) — only the *reservation* moves up to here.

Purpose: Satisfy the second half of Phase 4 and the entire authentication story. After this plan: a fresh visitor can hit `/`, see the public reel SDK-free; click "sign in"; redirect through Auth0 Universal Login; come back to `/app/`; get prompted to pick a handle (modal); pick one; navigate the app freely; and `/api/me` returns their user row from Postgres.

**JWKS middleware library choice: `jose`** wired into a small custom Hono middleware. Hono's built-in `hono/jwt` middleware accepts only fixed secrets, not JWKS endpoints. `jose` is the standards-track JWT library; its `createRemoteJWKSet(url)` returns a key resolver that caches keys and refreshes on rotation automatically. The custom middleware is ~30 lines and lives in `server/auth/jwt.ts`. AUTH-02 success criterion #4 ("rejects expired AND wrong-audience tokens") is verified by `server/auth/jwt.test.ts`, which mints both kinds of bad tokens against a stub JWKS — no live Auth0 tenant required for the test to pass.

**Frontend SDK: `@auth0/auth0-react`** — the official SDK. Provides `<Auth0Provider>`, `useAuth0()`, `loginWithRedirect()`, `getAccessTokenSilently()`. Locked because PROJECT.md says "Auth0 Universal Login" and the React SDK is the SDK Auth0 ships for SPAs.

**Lazy provisioning placement:** in a Hono middleware that runs immediately after `jwt.ts` on every `/api/*` route except `/api/health`. Single SELECT by auth0_sub; INSERT only when SELECT returns zero rows. The user row is attached to `c.var.user` so downstream handlers don't re-query.

**Handle picker as a modal**, not a separate route. Picking a handle is a UI mode, not a navigable destination. A modal blocks all `/app/*` routes uniformly without per-route guards. URL stays at whatever `/app/*` the user navigated to.

**Atomic commit invariant for the frontend auth wiring:** RequireAuth's body rewrite (which calls `useAuth0()`) AND the AppLayout provider stack mount (`<AuthProvider>`) MUST land in one commit. If split across two commits, the intermediate state has `useAuth0()` running outside its provider — `/app/*` would crash on every reload. Task 5 explicitly merges these two file changes into one task with one commit.

Output: full Auth0 round-trip working in dev; protected API; lazy user provisioning; handle reservation; modal picker; public reel still SDK-free.
</objective>

  **Create `server/auth/jwt.ts`** — the JWT validation middleware:
  ```typescript
  import { createRemoteJWKSet, jwtVerify, type JWTPayload } from 'jose';
  import type { Context, MiddlewareHandler } from 'hono';
  import { env } from '@server/env';

  // Auth0 issuer claim is the tenant URL with trailing slash.
  // JWKS lives at the well-known path; jose caches keys and refreshes on rotation.
  const ISSUER = `https://${env.AUTH0_DOMAIN}/`;
  const JWKS_URL = new URL(`https://${env.AUTH0_DOMAIN}/.well-known/jwks.json`);

  // Exported for test stubbing — the test replaces `jwksGetter` with a local
  // JWKS containing a freshly-generated keypair so it can mint expired and
  // wrong-audience JWTs without needing a live Auth0 tenant.
  export let jwksGetter: ReturnType<typeof createRemoteJWKSet> = createRemoteJWKSet(JWKS_URL, {
    cooldownDuration: 30_000,
    cacheMaxAge: 600_000,
  });
  export function __setJwksGetterForTest(getter: typeof jwksGetter) { jwksGetter = getter; }

  function bearer(c: Context): string | null {
    const h = c.req.header('authorization');
    if (!h) return null;
    const [scheme, token] = h.split(' ');
    if (scheme?.toLowerCase() !== 'bearer' || !token) return null;
    return token;
  }

  interface Auth0Payload extends JWTPayload {
    sub?: string;
    email?: string;
  }

  export const requireJwt: MiddlewareHandler = async (c, next) => {
    const token = bearer(c);
    if (!token) return c.json({ error: 'missing_bearer_token' }, 401);
    try {
      const { payload } = await jwtVerify(token, jwksGetter, {
        issuer: ISSUER,
        audience: env.AUTH0_AUDIENCE,
      });
      const p = payload as Auth0Payload;
      if (!p.sub) return c.json({ error: 'token_missing_sub' }, 401);
      c.set('auth0Sub', p.sub);
      c.set('auth0Email', p.email ?? '');
      await next();
      return;
    } catch (err) {
      // jose throws specific named errors (ERR_JWT_EXPIRED,
      // ERR_JWT_CLAIM_VALIDATION_FAILED, ERR_JWS_SIGNATURE_VERIFICATION_FAILED).
      // We collapse to a single 401 to avoid leaking which validation step
      // failed. Diagnostic detail goes to stderr per coding-style.md.
      const msg = err instanceof Error ? err.message : String(err);
      process.stderr.write(`JWT validation failed: ${msg}\n`);
      return c.json({ error: 'invalid_token' }, 401);
    }
  };
  ```

  **Why audience + issuer both checked, both required to match exactly:**
  - Audience prevents tokens minted for OTHER APIs in the same Auth0 tenant from being accepted here.
  - Issuer prevents tokens minted by a different Auth0 tenant from being accepted (defense against cross-tenant confusion).
  - jose's `jwtVerify` does both checks atomically and throws if either fails.

  ```typescript
  // The pattern is anchored — empty string and overlong inputs fail.
  // Hyphens are allowed but not at start/end (URL hygiene; nicer slugs).

    | { ok: true; handle: string }
    | { ok: false; reason: 'too_short' | 'too_long' | 'invalid_chars' | 'reserved' };

  // "lowercase-enforced") before checks so the user typing "Bryan" gets
  // "bryan" — the picker UI surfaces the lowercased form to make the
  // transformation visible.
  export function validateHandle(rawInput: string): HandleValidation {
  ```

  ```typescript
  import type { MiddlewareHandler } from 'hono';
  import { eq } from 'drizzle-orm';
  import { db } from '@server/db/client';
  import { users, type User } from '@server/db/schema';

  // AUTH-03: lazy user provisioning. Runs on every authenticated /api/*
  // call (mounted in server/index.ts after requireJwt). The first time a
  // user hits the API, this creates their users row keyed by Auth0's
  // immutable `sub` claim. Handle is left NULL — the picker UI fills it.
  //
  // Why middleware and not per-route: every authenticated handler needs
  // c.var.user. Doing it once here means handlers don't repeat the lookup.
  // Why not Auth0 webhook (the alternative considered in PROJECT.md
  // decisions table): webhook adds an external dependency, requires public
  // ingress, and creates a race where the user can hit the API before the
  // webhook fires. Lazy provisioning has neither problem.
  export const lazyProvisionUser: MiddlewareHandler = async (c, next) => {
    const auth0Sub = c.var.auth0Sub;
    const auth0Email = c.var.auth0Email;

    let user: User | undefined = await db.query.users.findFirst({
      where: eq(users.auth0Sub, auth0Sub),
    });

    if (!user) {
      const inserted = await db.insert(users).values({
        auth0Sub,
        email: auth0Email,
      }).returning();
      user = inserted[0];
      if (!user) {
        // Should be unreachable — INSERT ... RETURNING always returns the row
        // unless the row was rejected, in which case the INSERT itself throws.
        return c.json({ error: 'provisioning_failed' }, 500);
      }
    }

    c.set('user', user);
    await next();
  };
  ```

The second INSERT would fail on the `auth0Sub` UNIQUE constraint. For v1 single-user-flow this race is essentially nonexistent (a user only authenticates from one tab at a time during signup). If it ever bites, the fix is `INSERT ... ON CONFLICT (auth0_sub) DO UPDATE SET updated_at = now() RETURNING *` — Drizzle supports this via `.onConflictDoUpdate()`. Documented but not implemented now to keep the code minimal.</action>

  ```typescript
  // Responses:
  //   200 { handle } on success
  //   422 { error, reason } on pattern/reserved-word violation (defense-in-depth re-validation)
  //   409 { error: 'taken' } if another user has this handle
  //   409 { error: 'already_set', current: string } if THIS user already has a handle

    const candidate = typeof body?.handle === 'string' ? body.handle : '';

    if (!result.ok) {
      return c.json({ error: 'invalid_handle', reason: result.reason }, 422);

    // making this idempotent-ish (200 if same handle, 409 if different)
    // is more honest than letting the second POST silently win.
    const me = c.var.user;
    if (me.handle && me.handle !== result.handle) {
      return c.json({ error: 'already_set', current: me.handle }, 409);
    }
    if (me.handle === result.handle) {
      return c.json({ handle: me.handle });
    }

    // Try to claim. UNIQUE constraint on users.handle is the source of truth
    // for "is this taken?". A pre-check SELECT would race; we let the DB decide.
    try {
      const [updated] = await db.update(users)
        .set({ handle: result.handle, updatedAt: new Date() })
        .where(eq(users.id, me.id))
        .returning();
      if (!updated) return c.json({ error: 'update_failed' }, 500);
      return c.json({ handle: updated.handle });
    } catch (err) {
      // Postgres unique_violation = '23505'. The pg driver surfaces this as
      // err.code on the underlying error.
      const code = (err as { code?: string }).code;
      if (code === '23505') return c.json({ error: 'taken' }, 409);
      throw err; // any other failure → bubble to global error handler (Hono returns 500)
    }
  ```

  ```typescript
          {localValidation && !localValidation.ok && (
            <p className="text-xs text-amber-500">{errorFor(localValidation.reason)}</p>
          )}
          {typeof status === 'object' && 'error' in status && (
            <p className="text-xs text-amber-500">{status.error}</p>
          )}
          <button
            type="submit"
            disabled={status === 'submitting' || !localValidation?.ok}
            className="w-full bg-amber-500 text-black font-semibold py-2 rounded-lg disabled:opacity-50"
          >
            {status === 'submitting' ? 'Saving…' : 'Claim handle'}
          </button>
        </form>
      </div>
    );
  }

  function errorFor(reason: 'too_short' | 'too_long' | 'invalid_chars' | 'reserved'): string {
    switch (reason) {
      case 'too_short': return 'Too short (minimum 3 characters).';
      case 'too_long': return 'Too long (maximum 20 characters).';
      case 'invalid_chars': return 'Use lowercase letters, numbers, and hyphens only. Cannot start or end with a hyphen.';
      case 'reserved': return 'That word is reserved. Try another.';
    }
  }
  ```

  **Sub-step F — Create `src/auth/HandlePickerGate.tsx`** — checks user state on mount; renders the modal if needed. Wraps children. Uses AbortController to cancel the in-flight /api/me fetch on unmount (avoids React strict-mode double-fetch warnings and stale-state writes):
  ```typescript
  import { useEffect, useState, type ReactNode } from 'react';
  import { useApi } from '@/auth/useApi';
  import { HandlePickerModal } from '@/auth/HandlePickerModal';

  interface MeResponse {
    id: string;
    email: string;
    handle: string | null;
    createdAt: string;
  }

  // AUTH-07: prompt users without a handle on first authenticated visit.
  // Mounted INSIDE RequireAuth (which guarantees we're authenticated when
  // this runs). Fetches /api/me; if handle is null, renders the modal as
  // a sibling of children — children stay in the DOM (so the URL is
  // unchanged) but are visually obscured by the modal scrim.
  export function HandlePickerGate({ children }: { children: ReactNode }) {
    const api = useApi();
    const [me, setMe] = useState<MeResponse | null>(null);
    const [loaded, setLoaded] = useState(false);

    useEffect(() => {
      const ctrl = new AbortController();
      api('/api/me', { signal: ctrl.signal })
        .then(async (res) => {
          if (!res.ok) throw new Error(`me failed: ${res.status}`);
          const m = await res.json() as MeResponse;
          setMe(m);
          setLoaded(true);
        })
        .catch((err) => {
          // AbortError is expected on unmount — no state writes after abort.
          if ((err as { name?: string }).name !== 'AbortError') setLoaded(true);
        });
      return () => ctrl.abort();
    }, [api]);

    return (
      <>
        {children}
        {loaded && me && me.handle === null && (
          <HandlePickerModal onPicked={(handle) => setMe({ ...me, handle })} />
        )}
      </>
    );
  }
  ```
  Note: on first mount the modal does NOT appear until the /api/me round-trip completes — for a moment the user sees the AppLayout chrome with no modal. This is acceptable (sub-second) and avoids a flash of the modal when the user already has a handle.
