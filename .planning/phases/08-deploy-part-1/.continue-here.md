## Critical Anti-Patterns

| Pattern | Severity | Why it matters |
|---|---|---|
| Trusting cloud-init "Cloud-init complete." final_message | medium | The final_message fires even when intermediate modules errored. Phase 8 Wave 3 saw `package_update_upgrade_install` fail, runcmd continue, and the final message still print. Verify by `sudo cloud-init status --long` AND tail of `/var/log/cloud-init-output.log` for errors, not just the closing banner. |
| `ports:` lists in Docker Compose overrides | medium | Compose **concatenates** the `ports` key across `-f` files instead of replacing. `ports: []` in an override does NOT clear the inherited publish. To replace, use Compose 2.24+ `ports: !reset [...]` or remove the publish from the base file entirely. Don't assume override semantics on this key. |
| Building DATABASE_URLs via string interpolation of secrets | medium | `postgres://user:${POSTGRES_PASSWORD}@host/db` breaks silently when the password contains URL-unsafe characters (`/`, `+`, `=` from base64). Use URL-safe encoding (`openssl rand -hex` instead of `-base64`) or URL-percent-encode the password before interpolation. Zod `.url()` validation catches it at boot but only after the bad password is already committed to the postgres data volume. |

### F1.1 — nginx + certbot bootstrap chicken-egg on fresh VM (NEW, 2026-06-01)

`ops/nginx/timeline.conf` carries certbot's auto-injected TLS directives (committed back to the repo via the "Option A" pattern). On a fresh VM these reference files that only exist AFTER certbot has run:

  - `/etc/letsencrypt/options-ssl-nginx.conf`
  - `/etc/letsencrypt/ssl-dhparams.pem`
  - `/etc/letsencrypt/live/timeline.bryanlam.dev/fullchain.pem`

`nginx -t` fails before `certbot --nginx` can run. Chicken-and-egg.

**Workaround applied 2026-06-01:**

```bash
sudo certbot certonly --standalone -d timeline.bryanlam.dev --email ... --agree-tos --no-eff-email --non-interactive
sudo cp /usr/lib/python3/dist-packages/certbot_nginx/_internal/tls_configs/options-ssl-nginx.conf /etc/letsencrypt/options-ssl-nginx.conf
sudo openssl dhparam -out /etc/letsencrypt/ssl-dhparams.pem 2048  # ~1-2 min
sudo nginx -t  # passes now
sudo systemctl enable --now nginx
```

`certbot certonly --standalone` issues the cert without needing nginx (it runs its own temporary HTTP server on port 80). The `options-ssl-nginx.conf` and `ssl-dhparams.pem` files are NOT created by `--standalone` — only the `--nginx` plugin creates them. So we copy the template from `certbot_nginx`'s package and generate the dhparams manually.

**Fix paths (pick one for Phase 9 or 8.1.1 follow-up):**

- [ ] **Add a bootstrap section to cloud-init that pre-creates the two files.** Copy options-ssl-nginx.conf from the certbot_nginx package; generate ssl-dhparams.pem via `openssl dhparam`. Doing it in cloud-init means a fresh terraform apply produces a VM ready for `certbot --nginx` to "just work."
- [ ] **Document the bootstrap path explicitly in `infra/DEPLOY.md`.** The Nginx + Let's Encrypt section currently assumes `certbot --nginx` works on first run; reality is more nuanced when the repo's nginx config has the TLS directives baked in. Document `certonly --standalone` + template/dhparam pre-creation as the first-deploy path.
- [ ] **Reconsider the "commit certbot's edits back to repo" pattern.** The original intent was "repo as source of truth." But the bootstrap problem on fresh VMs is real. Alternative: keep `ops/nginx/timeline.conf` certbot-clean in the repo (no TLS directives) and let certbot inject them on the VM. Trade-off: the repo no longer reflects the actual on-disk nginx state.

**Verification when fixed:** From a fresh `terraform taint + apply`, walk through the Phase 8 Wave 2 nginx setup with no operator intervention beyond what DEPLOY.md prescribes. `nginx -t` should pass first try; `certbot --nginx` (not `--standalone`) should work end-to-end.

## F9 — Server-side `users.email` is empty for the test user (surfaced 2026-05-30)

`lazyProvisionUser` (`server/auth/lazyProvision.ts`) reads `c.var.auth0Email` and inserts it into `users.email`. The bryan test user's row shows `email = ''` (empty string, not NULL — the column is NOT NULL). Cause is that Auth0 access tokens don't carry the `email` claim by default — they have `sub` and `iat`/`exp` but not profile claims. The client-side `useAuth0()` hook reads from the ID token (which DOES carry email), so the HandlePicker fix from this phase works fine. But any server-side feature that needs the user's email (notifications, support escalation, account-recovery emails) currently has no data to work with.

- [ ] **Choose a fix path:**
  - **Auth0 Action to inject email into access token custom claims** (cleanest — declarative, runs server-side at token mint time): Auth0 Dashboard → Actions → Library → Custom → Login flow → set `api.accessToken.setCustomClaim('https://timeline.bryanlam.dev/email', event.user.email)`. Server reads from that custom claim instead of the standard `email` claim.
  - **Server calls Auth0 /userinfo endpoint** with the access token on first provision: adds one extra HTTP call per first-login but no Auth0 dashboard change.
  - **Server reads email from a separate ID-token-bearing endpoint** the client passes alongside the access token: more code change, but no Auth0 customization needed.
- [ ] **Backfill the empty email on existing rows** — once the fix is shipped, run `UPDATE users SET email = '<known-good>' WHERE auth0_sub = '<sub>'` for already-provisioned rows.
- [ ] **Consider widening the `email` column nullability** in `users` table — currently `NOT NULL`, but the empty-string workaround is worse than allowing NULL. Drizzle migration to relax the constraint.
