## Problem Statement

Revamp a 2025 course-project travel tracker (React-Leaflet + json-server + fake auth) into a portfolio-grade product that shows a recruiter three things in the first viewport on a phone: taste, technical chops, and the ability to ship something real. The bar is "Polarsteps, but the timeline plays like a movie you can export and share." The build target is a mobile-first web app with full auth and a seeded demo that plays without signup, plus a public per-user URL (`timeline.bryanlam.dev/u/bryan`) that can live on a resume.

The revamp is not a migration of the old code. The old repo at `~/Workspaces/timeline` is reference only for data shape (city model, reverse-geocode flow). Everything else is rewritten from scratch.

## What Makes This Cool

The map is not a lookup surface; it is the canvas, and time is the axis you scrub. A recruiter opens the link on their phone, sees a seeded reel of Bryan's actual trips already playing — camera flying between cities with pitch and bearing, photos popping in as the camera lands, captions fading, chapters ticking by like an Instagram Story made of geography. One tap produces an MP4 they can post to X or TikTok. Nobody in the category is treating the map this way because the tooling to do it well (MapLibre GL JS camera interpolation, vector tiles, headless rendering + FFmpeg) is only recently practical.

## Constraints

- **Audience:** hiring managers reviewing a portfolio link on a phone in ~30 seconds. No domain context, low patience, high visual discrimination.
- **Platform priority:** mobile-first, fully responsive. Desktop is the also-works surface.
- **Ship timeline:** 11 focused weekends, with 1 weekend of natural slack. No pre-build spike — discover MP4 render performance by hitting it in W9, not before.
- **Operator:** solo builder leveling up, evening/weekend cadence.
- **Existing infrastructure:** Auth0 tenant and an Oracle Cloud (OCI) VM already provisioned. App deploys via Docker Compose + Nginx + Let's Encrypt. Photos live in OCI Object Storage.
- **Domain:** `timeline.bryanlam.dev` (canonical, used everywhere).
- **VM sizing note:** OCI Always-Free VMs come in two shapes — AMD (1/8 OCPU, 1GB RAM) and ARM Ampere A1 (up to 4 OCPU, 24GB RAM). Puppeteer + MapLibre + FFmpeg needs the Ampere A1, minimum 2 OCPU / 8GB. Confirm current VM meets this before W8 deploy, resize or add a worker VM if not.

## Premises

1. **Portfolio artifact, not product.** Every feature must contribute to the 30-second wow or demonstrate a specific recruiter-facing skill (auth, CRUD, routing, upload, deploy). Nothing else ships in v1.
2. **Mobile-first, fully responsive.** Portrait phone is the primary canvas. Touch gestures (tap-hold scrub, swipe chapters) are signature, with `prefers-reduced-motion` fallback mandatory, not optional.
3. **MapLibre GL JS, not Leaflet.** The cinematic thesis cannot execute on a DOM renderer.
4. **Seed the hero.** Landing page is the seeded `demo` user's reel already playing, no signup wall.
5. **Existing infrastructure wins.** Auth0 for identity, OCI VM for hosting, Postgres in Docker, OCI Object Storage for photos.
6. **MP4 export is a stretch goal in v1, not a launch gate.** Build order tries the fallback ladder in W10-11: server-side Puppeteer + FFmpeg first; if too slow (>90s wall-clock for 300 frames), drop to client-side `MediaRecorder` + canvas capture (10s, 720p, iOS 17+ floor); if that fails on target hardware, drop to a 10s looping GIF export. **If all three fail, MP4 is cut from v1 and shipped as a v2 feature.** The launch (W12) does not block on MP4 working. Success criteria reflect this — MP4 is "ships if any ladder rung works" instead of mandatory.
7. **Handles are user-chosen strings** (`[a-z0-9-]{3,20}`, case-insensitive, uniqueness-enforced, reserved-word list). Auth0 user IDs are not exposed in URLs.
8. **Gesture model is validated inline during W1-2, not as a separate spike.** End-of-W1 acceptance: hardcoded reel passes the 30-second test on a real iPhone with the full gesture state machine wired. If transitions break, fix in the prototype before W2 begins. Cheaper to rework a prototype than to spike-then-rebuild.

## Approaches Considered

### Approach A: The Reel (minimal viable)
Reel-only, no dashboard, no CRUD UI beyond a tiny modal. 4-6 weekends. Rejected — doesn't demonstrate enough recruiter-facing surface.

### Approach B: The Atlas (hybrid app, chosen)
Landing = seeded reel. Authenticated `/app` with Reel (hero) + a single "Trips" list view (combining Map pins and chronological entries into one scrollable view, not two separate tabs — scope cut). Public `/u/:handle`. MP4 export with fallback.

### Approach C: The Reel Engine (creative tool)
Paste JSON → shareable URL. 3-5 weekends. Rejected — reads as "tool" not "product."

## Recommended Approach

**Approach B — The Atlas, scope-tightened after review.**

Key post-review changes from rev 1:
- **Three views collapsed to two.** Reel (hero) + Trips (combined map + chronological list, not two tabs). Cuts one weekend of polish work and removes redundant surfaces.
- **"Trips" as a separate entity is cut.** v1 schema is flat cities (`user_id`, `order`, optional `trip_label` string). Trip-grouping can be a v2 enhancement.
- **MP4 pipeline has a pre-committed fallback ladder** triggered by the W10 server-render benchmark.
- **Deploy gets 2 weekends, not 1.** First-time OCI + Docker Compose + TLS + CI is not a one-weekend task.
- **Schema, photo pipeline, gesture model, and accessibility plan are specified below, not deferred to "later."**

### Data schema (v1, final)

```
users (
  id            uuid PK,
  auth0_sub     text UNIQUE NOT NULL,  -- Auth0 subject claim
  handle        text UNIQUE NOT NULL,  -- lowercase, [a-z0-9-]{3,20}
  display_name  text,
  created_at    timestamptz
)

cities (
  id            uuid PK,
  user_id       uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name          text NOT NULL,
  country       text,
  country_code  text,
  lat           double precision NOT NULL,
  lng           double precision NOT NULL,
  visited_at    date NOT NULL,
  notes         text,
  order_index   integer NOT NULL,      -- reel chapter order, user-reorderable
  trip_label    text,                  -- optional free-text grouping ("Japan 2024")
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, order_index) DEFERRABLE INITIALLY DEFERRED  -- allow reorder in txn
)

photos (
  id            uuid PK,
  city_id       uuid NOT NULL REFERENCES cities(id) ON DELETE CASCADE,
  object_key    text NOT NULL,         -- OCI Object Storage key
  thumb_key     text NOT NULL,         -- 400px thumbnail key
  width         integer,
  height        integer,
  order_index   integer NOT NULL,
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now()
)

notifications (
  id            uuid PK,
  user_id       uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  kind          text NOT NULL,         -- 'mp4_ready' | 'mp4_failed'
  payload       jsonb NOT NULL,        -- { job_id, download_url?, error_code? }
  read_at       timestamptz,
  created_at    timestamptz NOT NULL DEFAULT now()
)

reserved_handles (handle text PK)      -- seeded with 'admin', 'api', 'demo', 'u', 'login', 'logout', 'settings', 'about'
```

Indexes: `cities(user_id, order_index)`, `photos(city_id, order_index)`, `notifications(user_id, created_at DESC) WHERE read_at IS NULL`. Unique indexes from `UNIQUE` constraints: `users(handle)`, `users(auth0_sub)`.

FK cascades deliberately chosen over soft-delete — v1 is simpler, user-delete is an intentional full-erase (GDPR-friendly).

**Reorder-flow constraint:** the `DEFERRABLE INITIALLY DEFERRED` on `cities(user_id, order_index)` only relaxes inside an explicit transaction. The reorder endpoint must wrap all position swaps in `BEGIN/COMMIT` (Drizzle's `db.transaction()`). Outside a transaction, any intermediate collision raises immediately. This is documented at the endpoint implementation in W5.

### Photo pipeline

- **Client:** accept JPEG/PNG/HEIC. Convert HEIC → JPEG client-side via `heic2any` (iPhone default format).
- **Resize:** scale to max 2048px longest edge (canvas `drawImage`), JPEG quality 0.85.
- **Thumbnail:** generate 400px square-crop thumbnail client-side in same pass.
- **EXIF:** strip all EXIF client-side (PII: GPS coords, camera serial, timestamps). User-entered lat/lng comes from the map picker, not EXIF.
- **Upload:** mint an OCI Pre-Authenticated Request (PAR) URL from the backend (scoped write, 10min TTL, one object). Frontend PUTs directly. Repeat for thumbnail.
- **Max:** 10 photos per city in v1; 5MB per file post-resize hard limit.

### Mobile gesture state machine (load-bearing — v1 ships or dies on this)

The reel is **full-bleed**, scroll is **disabled** on the reel surface (`html { overflow: hidden }` scoped via route). All vertical motion is interpreted by the gesture handler, not the browser.

```
States: IDLE | SCRUBBING | CHAPTER_SWIPE | MAP_INTERACT | PAUSED | SUSPENDED

Input priority (highest first):
1. Three-finger or accessibility gesture     → yield to OS (do not intercept, no preventDefault)
2. Two-finger (pinch/rotate/drag)            → MAP_INTERACT, suspends auto-reel 3s after release
3. Long-press ≥200ms, single-finger          → SCRUBBING, horizontal drag drives time
4. Vertical flick ≥30px in <300ms            → CHAPTER_SWIPE (up = next, down = prev)
5. Single tap on caption/photo overlay       → open detail sheet (consumed by child)
6. Single tap on reel background             → toggle PAUSED / IDLE
7. No input                                  → IDLE (auto-play)

Transition rules (complete, not priority-only):
- SCRUBBING + second finger lands  → cancel scrub, promote to MAP_INTERACT.
- SCRUBBING + finger leaves screen → commit scrub position, return to IDLE (auto-play from new time).
- CHAPTER_SWIPE → IDLE after 400ms flyTo animation completes.
- MAP_INTERACT + 3s idle           → IDLE (auto-reel resumes from current time).
- Any state + `visibilitychange` to hidden → SUSPENDED (pause reel, stop map animation).
- SUSPENDED + `visibilitychange` to visible → IDLE (resume from paused time, do not auto-scrub).
- Any state + `orientationchange` → SUSPENDED, then IDLE after 300ms (let layout settle, resize canvas).

Platform-specific rules:
- iOS Safari pull-to-refresh: disabled via `body { overscroll-behavior: none }` on reel route.
- iOS back-edge-swipe (left 20px): NOT intercepted — user expectation preserved, means left-edge scrub is unavailable. Document as tradeoff.
- iOS 3-finger tap (accessibility): priority 1 above.
- Android hardware back button (`popstate`): treat as "close detail sheet if open, else navigate /app". Explicit `history.pushState` on sheet open.
- Double-tap-to-zoom: disabled via `touch-action: manipulation`.
- Browser bottom-bar shrink on iOS: reel sizes to `100dvh` (dynamic viewport) with a `svh` fallback.
- MapLibre's own single-finger drag: disabled (`dragPan: false, touchZoomRotate: false` on reel), re-enabled only in MAP_INTERACT.
- Keyboard (desktop): Space = play/pause, ←/→ = scrub ±1s, ↑/↓ = chapter prev/next, Esc = close sheet.
```

### Accessibility and reduced-motion fallback

- **`prefers-reduced-motion: reduce`** triggers a static-chapter view: vertical list of city cards with photo + caption + date, scroll-native, no camera animation. Same data, no motion, Lighthouse-clean.
- **Keyboard controls** for the reel: Left/Right = scrub 1s, Up/Down = chapter, Space = play/pause, Enter = open detail.
- **ARIA:** reel container is `role="region"` with `aria-label` (not `role="application"`, which suppresses native AT navigation and is a known footgun); chapter transitions fire an `aria-live="polite"` announcement ("Kyoto, October 2024"); photo overlays have `alt` from user-entered captions (empty-alt if no caption — decorative).
- **Focus trap** in detail sheet, `Esc` closes.
- **Color contrast** audited to WCAG AA on overlay text against worst-case (bright photo) backgrounds; overlay has a gradient scrim.

### Empty and error states (specified, not deferred)

- **Public reel, user has 0 cities:** map shows world view with a single caption "No trips yet. Check back soon." (Marketing-friendly — no broken UI.)
- **Public reel, user has 1 city:** no inter-city flyover; camera orbits slowly around the single city with photos cycling. Orbit camera spec: 8s full orbit at zoom 14, pitch 60.
- **Public reel, user has 2+ identical coordinates:** sequential identical chapters collapse to one "chapter group" with multiple photos, no redundant flyTo.
- **Authenticated /app, 0 cities:** onboarding card: "Add your first city" → opens add-city sheet.
- **Photo upload fails:** inline retry with exponential backoff (max 3 retries), "Upload failed, tap to retry."
- **MP4 render fails:** notification card with error category (queue timeout, render error, storage error) + retry button. Never silently drop a job.
- **MapTiler rate limit hit:** swap to OSM raster fallback tile URL with a banner: "Map service limited; some detail reduced." Degrade gracefully.
- **Offline:** out of scope for v1 (no service worker). Network-required. Cut to preserve the weekend budget.

### Auth0 + public-route strategy

- Two route groups: public (`/`, `/u/:handle`, `/login`) and private (`/app/*`).
- `<Auth0Provider>` wraps ONLY the private tree. Public routes render without SDK init — no silent-auth calls, no third-party-cookie dependency, no redirect flashes on iOS Safari.
- Public reel data is fetched via unauthenticated GET (`/api/public/reels/:handle`), cached edge-style in Nginx for 60s.
- JWT middleware on `/api/*` protected routes only; public routes (`/api/public/*`) have no middleware.
- **User provisioning: lazy.** JWT middleware checks `users` for a row matching the token's `sub` claim; if absent, creates one (auth0_sub, email, no handle yet) in the same request, then proceeds. First `/app` load redirects to a one-time handle-picker if `handle IS NULL`. No Auth0 webhook surface, no public provisioning endpoint.
- **Auth0 connections in v1:** Google social + email/password (Auth0 Database connection). Confirm both sit inside the Free tier 25k MAU bucket.

### Public photo access

- OCI Object Storage bucket has two prefixes: `public/` (Read accessible to anonymous) and `private/` (auth-only). Bucket policy is set during W8 deploy.
- All photos uploaded for a city land in `public/{user_id}/{photo_id}.jpg` and `public/{user_id}/{photo_id}.thumb.jpg` — they will be visible on the public reel anyway, so there is no security delta.
- Reel data API (`/api/public/reels/:handle`) returns absolute photo URLs (`https://objectstorage.{region}.oraclecloud.com/.../public/...`), no per-request signing.
- **Orphan cleanup:** photo upload uses 10-minute write-PAR; if user uploads but never saves the city, object is orphaned. Daily cron (`docker compose run --rm api npm run gc:photos`) lists `public/{user_id}/*` keys and deletes any not referenced in `photos.object_key` or `photos.thumb_key`. v1 acceptable; can swap for queue-driven later.

### API surface (v1, locked)

```
PUBLIC (no auth):
  GET    /api/public/reels/:handle               → { user, cities[], photos[] } in one shot, 60s Nginx cache
  GET    /api/public/handles/:handle/exists      → for handle picker preview during signup

AUTH (JWT required, lazy-provisioned on first call):
  GET    /api/me                                  → current user row, 401 if missing handle
  POST   /api/me/handle { handle }                → claim handle, returns 409 if taken/reserved

  GET    /api/cities                              → my cities, ordered by order_index
  POST   /api/cities { ...city }                  → create
  PATCH  /api/cities/:id { ...city }              → update (NOT order_index — use reorder)
  DELETE /api/cities/:id                          → delete (cascades photos)
  PATCH  /api/cities/reorder { items: [{id, order_index}] }  → bulk, single transaction

  POST   /api/cities/:id/photos/upload-url        → mint 10-min write-PAR for one photo + thumb
  POST   /api/cities/:id/photos { object_key, thumb_key, w, h }  → register uploaded photo
  DELETE /api/photos/:id                          → delete row + Object Storage object

  POST   /api/exports                             → enqueue MP4 render (server path) OR signal client to record
  GET    /api/notifications?since=:ts             → poll for completed exports (server path only)
```

The reorder endpoint MUST wrap all swaps in `db.transaction()` so the deferred-unique constraint relaxes correctly.

### MP4 pipeline (fallback ladder, benchmark in W10)

**Primary path (server-side):**
- BullMQ job queue + Redis container.
- Worker container runs headless Chromium (Puppeteer) with `--use-gl=swiftshader` on ARM Ampere (no GPU available, so ANGLE/Vulkan flags would land on software rendering anyway — SwiftShader is the honest setting).
- Capture loop: tick MapLibre camera at target framerate, `page.screenshot()` each frame, pipe through `ffmpeg -framerate 30 -i frame-%05d.png -c:v libx264 -preset fast -crf 23 out.mp4`.
- Upload to Object Storage, create a time-limited PAR for download, insert notification row.
- Frontend polls `/api/notifications?since=:ts` on a 5s interval while a render is active, stops after completion or cancellation.
- Concurrency: `BULL_CONCURRENCY=1`. RAM math at 8GB: Chromium headless ≈1GB + FFmpeg ≈500MB + Node API ≈200MB + Postgres ≈500MB + Redis ≈100MB + OS overhead ≈1GB ≈ 3.3GB at idle. A second concurrent Chromium would land near OOM under spike. Rate limit + concurrency-1 is the safe v1 posture; scale by adding a dedicated worker VM in v2 if demand warrants.
- Per-user rate limit: 5 renders per 24h (DB-enforced, return 429 with retry-after).
- Timeout: 5 minutes per job, auto-fail with clear error.

**Fallback path (client-side, triggered if W10 server-render benchmark fails — see W10 step for threshold):**
- `canvas.captureStream(30)` on MapLibre WebGL canvas, `MediaRecorder` with explicit capability probing.
- **Minimum platform floor:** iOS 17+ / Safari 17+ / Chrome 109+ / Firefox 115+. iOS 16 and earlier do not reliably support `MediaRecorder` + WebGL `captureStream` (black-frame bug). Users below floor see "Export unavailable on this browser — try the latest Safari, Chrome, or Firefox" with the platform-checker running on page load.
- **Codec order:** probe `video/mp4;codecs=avc1` → `video/webm;codecs=vp9` → `video/webm;codecs=vp8`. First supported wins.
- **Tertiary fallback (if even WebM fails):** "Record your screen" instructions deep-linked to iOS screen-record settings. Not auto — user-initiated. Better than a dead button.
- Render length capped at 10 seconds (capture-during-playback).
- Download via `URL.createObjectURL` + anchor click. No server, no queue, no rate limit — browser-local.
- UX tradeoff: recording happens live during playback, so user waits the clip's real duration. Acceptable at 10s.

**Decision: which path ships** is made during W10 based on the in-flight benchmark. Threshold: "W10 server-render ≤90s for 300 frames → server path ships; W10 server-render >90s → drop to client fallback in W11; if client probes fail on iOS 17 Safari → drop to GIF; if all three fail → cut MP4 to v2."
