### Auto-fixed Issues

**1. [Rule 1 - Bug] RESEARCH Pattern 5's `identity_domains[0].url` data source path is wrong against `oracle/oci v6.37.0`**

- **Found during:** Task 1 (initial `terraform validate` after copying Pattern 5 verbatim)
- **Issue:** Pattern 5 wrote `data.oci_identity_domains.default.identity_domains[0].url`. Schema dump via `terraform providers schema -json` (against a clean `/tmp/tfschema` module with `oracle/oci ~> 6.0`) shows the data source's list-of-objects attribute is named `domains`, not `identity_domains`. `terraform validate` initially failed with "Unsupported attribute" on lines 63 and 84 (the two `oci_identity_domains_user.*` resources' `idcs_endpoint` references).
- **Fix:** Replaced both occurrences plus the trust resource's `idcs_endpoint` reference (3 total) with `data.oci_identity_domains.default.domains[0].url`. Added an inline comment on the first one citing RESEARCH Assumption A3/A10 and the schema-dump verification step.
- **Files modified:** `infra/terraform/iam.tf`
- **Verification:** `terraform validate` now passes; `terraform fmt -check` still clean.
- **Committed in:** `f44a662`

**2. [Rule 1 - Bug] `allowed_token_issuers` nested block on `oci_identity_domains_identity_propagation_trust` does not exist in `oracle/oci v6.37.0`**

- **Found during:** Task 1 (second `terraform validate` after fixing the `domains[]` path)
- **Issue:** RESEARCH Pattern 5 wrote `allowed_token_issuers { value = "https://token.actions.githubusercontent.com" }` as a nested block on the trust resource. Schema dump shows no such block — the only nested blocks on this resource type in v6.37.0 are `impersonation_service_users`, `keytab`, `tags`, and `timeouts`. The top-level `issuer` attribute (which the trust HAS) is the documented JWT-issuer constraint; the `allowed_token_issuers` block is either older-provider syntax or planned-but-not-shipped.
- **Fix:** Removed the `allowed_token_issuers { ... }` block. Added an inline NOTE comment explaining the field-name verification and that `issuer` + `impersonation_service_users.rule` together provide the JWT issuer constraint + subject-claim narrowing.
- **Files modified:** `infra/terraform/iam.tf`
- **Verification:** `terraform validate` passes.
- **Committed in:** `f44a662`

**3. [Rule 1 - Bug] `lifecycle { ignore_changes = [defined_tags] }` on Identity Domains (IDCS) resources fails because they don't expose `defined_tags`**

- **Found during:** Task 1 (would-be third `terraform validate` — caught proactively after Deviation #2 by inspecting the schema dump for the user + trust resource types)
- **Issue:** Plan said "lifecycle ignore_changes = [defined_tags] on every NEW OCI resource introduced" (per the worktree pre-context Pitfall-8 reference and the action's HCL excerpts). Schema dump shows neither `oci_identity_domains_user` nor `oci_identity_domains_identity_propagation_trust` exposes a `defined_tags` attribute — IDCS resources use SCIM `tags` blocks (different shape: `tags { key = ..., value = ... }`) rather than the OCI Native API's `defined_tags = { "namespace.key" = "value" }` map. `terraform validate` would reject `ignore_changes = [defined_tags]` on these resources.
- **Fix:** Removed the lifecycle blocks from the two `oci_identity_domains_user.*` resources and the `oci_identity_domains_identity_propagation_trust.github_actions` resource. Kept the lifecycle blocks on `oci_identity_policy.gha_deployer_manage` + `oci_identity_policy.gha_pr_reader_inspect` (those ARE OCI Native API resources with the `defined_tags` map attribute — schema confirmed). Added an inline NOTE comment on `gha_deployer` documenting the asymmetry.
- **Files modified:** `infra/terraform/iam.tf`
- **Verification:** `terraform validate` passes (already covered by the same run that caught Deviation #2).
- **Committed in:** `f44a662`

**4. [Rule 1 - Bug] Negative grep `! grep -q 'oci_identity_identity_provider' iam.tf` matched explanatory comment text (same `feedback_grep_guard_vs_comments.md` landmine that bit Plan 02 twice)**

- **Found during:** Task 1 verify run
- **Issue:** Initial draft of the header comment block above the OIDC resources said `do NOT use the legacy \`oci_identity_identity_provider\` resource (SAML federation)`. The verify check `! grep -q 'oci_identity_identity_provider' iam.tf` (testing the absence of the legacy resource declaration) matched the literal substring inside the comment, even though no actual resource declaration exists. Same class of false-positive as Plan 02 Deviations #1 and #2 — substring match cannot distinguish HCL declarations from explanatory prose.
- **Fix:** Paraphrased the comment to "do NOT use the legacy SAML-federation resource from the v5-era IAM provider (the one whose type omits `_domains_` and ends in `_provider`)." — preserves the guidance, removes the literal substring. Targeted verification: `grep -E '^resource "oci_identity_identity_provider"' iam.tf` returns no matches (the actual semantic check the negative grep was trying to enforce).
- **Files modified:** `infra/terraform/iam.tf`
- **Verification:** `! grep -q 'oci_identity_identity_provider' iam.tf` now passes literally; targeted resource-declaration grep confirms no legacy resource exists.
- **Committed in:** `f44a662`

## Issues Encountered

- **`terraform providers schema -json` requires backend init when a backend is configured.** The project's `backend.tf` is the S3 backend (configured by Plan 01); `-backend=false` works for `init` and `validate` but the `providers schema` subcommand refuses to run. Workaround: created an isolated `/tmp/tfschema/main.tf` module declaring only the `oracle/oci ~> 6.0` provider (no backend), ran `terraform init -input=false` there, then `terraform providers schema -json > /tmp/schema.json` (5.7 MB output). Parsed with Python to introspect the relevant resource + data-source schemas. This is a generally useful local-development workflow for any future TF stack that needs schema introspection without touching state.

- **No documentation lookup proved necessary beyond the schema dump.** Tried `npx ctx7 docs /websites/oracle_en-us_iaas_content` for the data source attribute name (Assumption A3) but the highest-relevance results were `oci_identity_availability_domains` (the wrong data source — collision on the prefix). The schema dump is the canonical source of truth for terraform provider field names; documentation pages can lag schema reality, especially on Identity Domains / IDCS resources where Oracle has multiple co-existing provider generations.

- **actionlint was not pre-installed locally** — installed via `brew install actionlint` (v1.7.12). DEPLOY.md Section 1 already lists it as recommended-but-optional. Workflow passed lint on the first try (no warnings, no errors).

## Threat Surface Scan

No new security surface introduced beyond what the plan's `<threat_model>` already documents. Mitigation status:

- **T-08.1-05 (auto-apply without manual gate):** mitigated — `apply` job has `environment: production` + `concurrency: { group: terraform-apply, cancel-in-progress: false }`. Per worktree pre-context, the `production` GitHub Environment is already configured with `usbryanchlam` as required reviewer + `main` branch restriction.

- **T-08.1-04-pr-creep (PR identity privilege escalation):** mitigated — `gha_pr_reader_inspect` policy grants ONLY `inspect all-resources` + `read object-family`. NEVER `manage` (negative grep `! grep -E 'gha_pr_reader.*manage all-resources|gha-pr-reader.*manage all-resources' iam.tf` passes).
- **T-08.1-fork (fork PR spoofing):** accepted — GitHub doesn't issue OIDC tokens to fork PRs by default; DEPLOY.md Section 4 now documents the limitation (Pitfall 10).
- **T-08.1-cron-secret (drift cron OIDC + state-backend SK exposure):** accepted — drift run inherits the same env vars as `plan` job (PR-reader OIDC identity, read-only OCI scope; state-backend SK for state read). Plan output in GH Issues exposes OCIDs but not secrets.
- **T-08.1-upst-replay (UPST reuse across jobs):** mitigated — `plan` and `apply` jobs each do their OWN OIDC token exchange (no token persisted across jobs); `~/.oci/upst.token` is written per-job onto the ephemeral runner VM and dies with the runner.
